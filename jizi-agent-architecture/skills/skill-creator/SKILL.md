# Skill 创建与复写

## 具体能力

- 设计 Skill 触发描述
- 精炼做事方法
- 拆分复杂文档
- 规划资源结构
- 校验 Skill 质量

## 做事方法

把 Skill 当作给模型看的专业做事手册，而不是普通说明文档。默认模型已经足够聪明，只补充它无法稳定知道的流程、约束、领域规则、工具用法和输出要求。

创建或复写 Skill 时，先确认它解决什么任务、何时应该触发、哪些信息必须进入正文。名称要短、具体、可识别；描述要说明触发场景和任务价值；能力列表保留 3-8 条，每条都应是可执行能力。

正文只保留真正会影响执行质量的内容：

- 工作流程和决策规则。
- 容易出错的边界条件。
- 必要的校验步骤。
- 输出结构和交付标准。
- 必须使用或避免的工具、依赖、格式和路径。

删除寒暄、安装日志、更新记录、过长背景、重复解释、营销措辞和模型已显而易见的常识。示例只在能防止误判时保留，并保持短小。

如果一个文档包含多个可独立触发、独立维护、独立组合的职责，要拆成多个 Skill。不要因为它们同属一个领域就强行合并。只有所有规则每次都必须同时执行、拆开会失去意义时，才保留为一个 Skill。

为复杂 Skill 规划资源时遵循渐进加载：

- 核心流程写在 `SKILL.md`。
- 长篇规范、表结构、API 细节放进 references。
- 需要稳定重复执行的代码放进 scripts。
- 输出会用到的模板、图片、字体或样例放进 assets。

不要创建无关文档，例如 README、安装指南、更新日志或快速参考。Skill 目录只保留会直接提升执行质量的文件。

质量检查：

- 是否能从描述判断何时触发。
- 正文是否足够短，且没有稀释重点。
- 每条能力是否可执行。
- 是否包含必要边界和输出要求。
- 是否把长资料拆到合适资源中。
- 是否避免重复、空泛和过度解释。
