## A. 交接阅读顺序与迁移口径

### A.1 新模型优先阅读顺序
给下一个模型接手时，建议按这个顺序读，不要一上来直接翻代码：

1. `交接注意事项.md`：先读这个。它是入口地图，告诉它项目怎么迁移、哪些东西别上传、当前做到哪、还欠什么。
2. `docs/审计待办-2026-07-09.md`：再读这个。它是当前还没处理的风险和工程待办，适合直接排下一轮。
3. `开发清单.md`：需要了解“为什么做这个软件、功能总账、版本变更”时读。
4. `施工清单.md`：需要继续写代码、查某个功能怎么落地时读。
5. `技术评审报告.md`：需要做安全/架构复审时读。
6. `全期开发进度统计.md` / `软件开发全记录.md`：需要快速理解项目历史和整体完成度时读。

### A.2 GitHub 迁移口径
当前目录已经能跑 `git status`，但仓库处在刚开始管理的状态，文件基本还是未跟踪。交给新模型时要提醒它：

- 先看 `.gitignore`，确认不会把运行数据、密钥、编译产物传上去。
- 第一次提交前先跑 `git status --short`，确认哪些文件会进仓库。
- 不要上传 `data/` 里的私人配置和 key；key 应该由用户在应用内配置。
- 不要上传 `outputs/`、`logs/`、`app/node_modules/`、`app/dist/`、`app/src-tauri/target/`、`app/src-tauri/python-dist/`、`python/venv/`。

### A.3 源码仓库建议包含
- `app/`：前端、Tauri、Rust 桌面端源码。
- `python/`：Python 工具服务源码、测试、依赖清单；不含 `python/venv/`。
- `jizi-agent-architecture/`：姬子人格与 Skill 架构文件。
- `docs/`：审计待办、模块化规范、错误总结等辅助文档。

- `开发清单.md`、`施工清单.md`、`交接注意事项.md`、`技术评审报告.md`、`全期开发进度统计.md`、`软件开发全记录.md`。

### A.4 构建/运行产物建议排除
这些东西可以重新生成，不适合进 GitHub：

- `app/node_modules/`
- `app/dist/`
- `app/src-tauri/target/`
- `app/src-tauri/python-dist/`：发布前用 `node python/build-dist.mjs --sync-only` 或完整 `node python/build-dist.mjs` 重新生成。
- `python/venv/`
- `data/`、`outputs/`、`logs/`：里面可能有私人数据、API Key、运行历史或产物。

### A.5 新机器基础环境
- Node.js：当前使用 npm 11.x，较新的 LTS/Current 版本即可。
- Rust：`app/src-tauri/Cargo.toml` 最低 `rust-version = "1.77.2"`，建议用 rustup 安装稳定版。
- Python：Python 3.12 系列；按 `python/requirements.txt` 安装依赖。
- Windows WebView2 Runtime：Tauri 桌面应用需要。

## E. 继续开发时的关键约束

- 遵循“主干 + 功能节点”低耦合：新增功能尽量放独立文件或目录，主文件只做接线。
- 不要把运行状态写回原画布；运行使用只读快照，原画布保持干净。
- 失败隔离原则不变：某节点失败，只影响依赖它的下游，不应阻断无依赖的并行分支。
- 报告中心只采集已结束运行；运行中画布的部分完成节点不提前进入统计。
- 总 Agent 会话隔离：一个会话的发送/取消状态不能影响其他会话。
- 姬子系统提示词仍是文件导入模式，不要擅自恢复手动编辑。
- **prompt caching 依赖稳定前缀**：`chat()` 的 `system` 只放人格块（每轮字节不变才命中缓存）；当前时间、记忆、搜索、摘要等每轮变动的内容必须放 `text`，不要塞进 `system`，否则会击穿缓存。
- 新增外部 HTTP 调用时，需要同步 Tauri plugin-http、Rust 插件注册、capabilities 白名单，不能用通配符放行。

## H. 正式安装包构建步骤（v2.90）

产出**标准 Windows 安装包**（NSIS，currentUser 安装），随包分发一个可重定位的独立 CPython，无需目标机预装 Python。

### H.1 为什么这样设计
- **currentUser 安装**：装到 `%LOCALAPPDATA%\Programs\multi-agent-tool`（类似 VS Code），无 UAC、安装目录用户可写。这一点是关键——应用是「绿色便携」架构，`data/`（含明文 API Key）、`outputs/`、`logs/`、`python/` 全在 exe 同级且**运行时要写**（姬子往 `python/tools/custom/` 写生成工具、`installer.py` 用 pip 装依赖）。装到只读 Program Files 会破坏这些功能；currentUser 可写使架构原样成立。
- **随包带 Python**：开发 `python/venv/` 的 `pyvenv.cfg` 硬编码开发机绝对路径，拷别机即失效；姬子动态造工具又依赖运行时 `pip install`，PyInstaller 冻结会破坏它。故用 **python-build-standalone**（可重定位、自带 pip 的完整 CPython 3.12）。

### H.2 构建步骤（需联网，耗时几分钟）
```bash
# 1. 生成可分发 Python 环境（下载 ~44MB standalone CPython + pip 装依赖，产出 app/src-tauri/python-dist/，约 261MB）
cd /c/Users/Admin/Desktop/agent-collab
node python/build-dist.mjs

# 2. 构建安装包（前端 build + Tauri 打包，python-dist 作为资源打进包）
cd app
npm run tauri:build

# 3. 产物位置
#    app/src-tauri/target/release/bundle/nsis/multi-agent-tool_<版本>_x64-setup.exe
```

### H.3 落点校验（安装后）
安装目录内应有：`python/runtime/python.exe`、`python/app.py`、`python/tools/`。运行时 `python_manager.rs` 的 `find_python_exe()` release 分支按 `python/runtime/python.exe` 定位，与 `find_project_root()`（exe 同级）一致。若 Tauri 把资源落到非 exe 同级目录，需改用 `app.path().resource_dir()` 兜底（本次实测落点正确，未启用兜底）。

### H.4 注意
- **未签名**：无代码签名证书，首次运行可能触发 Windows SmartScreen「未知发布者」，需点「仍要运行」。签名超出本次范围。
- **体积**：NSIS 包预计 ~120–180MB（standalone CPython + PyMuPDF/pdfplumber/pillow/cryptography 等重依赖）。这是「离线开箱即用 + 保留运行时 pip 造工具」的代价；日后要瘦身可改「首次运行联网下载 Python」。
- **构建机需联网**：`build-dist` 要从 GitHub 下载 standalone Python；下载后离线可重复 `tauri:build`。
- **Python 版本**：standalone 取 3.12（对齐开发 venv 3.12.8），避免 site-packages ABI 不匹配。
- **改动清单**：新增 `python/build-dist.mjs`；改 `app/src-tauri/tauri.conf.json`（bundle.resources + nsis.installMode + targets=nsis）、`app/src-tauri/src/python_manager.rs`（仅 `find_python_exe` release 子路径）。`storage.rs`/`lib.rs` 数据日志根、`dynamic.py`/`installer.py`、前后端版本握手常量**均未改**（currentUser 可写使其成立）。
