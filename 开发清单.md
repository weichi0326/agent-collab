## 一、项目概述

### 1.1 产品定位
- **类型**：多 Agent 协同工作流工具
- **运行平台**：桌面应用（仅 Windows）
- **目标用户**：游戏测试人员（初始用户，个人使用）
- **核心价值**：通过可视化画布编排多个 AI Agent，自动化完成复杂任务流程

### 1.2 核心场景
用户通过拖拽 Agent 到画布并连线，实现：
- Agent 1（分析策划需求）→ 输出需求点 → Agent 2（写测试用例）→ 输出测试用例
- 支持单 Agent 启动、顺序启动、并行启动
- 每个 Agent 独立运行，结果本地存储

---

## 二、核心概念模型

### 2.1 层级结构
```
项目
 ├─ LLM 配置（全局，可配置多个模型）
 ├─ 工具集（标签化，可动态添加/删除）
 ├─ Agent 库（可创建多个 Agent，每个选用一个 LLM + 工具标签）
 ├─ 画布（多个画布，每个画布包含多个 Agent 节点 + 连线）
 └─ 总 Agent（自然语言控制画布和 Agent）
```

### 2.2 关键定义

#### Tool（工具）
- **最小执行单元**，只执行一件事
- **标签化**：一个工具 = 一个标签（如 `excel` 标签代表所有 Excel 操作能力）
- **实现方式**：Python 脚本（优先三方库，其次 GitHub 项目，最后自写）
- **可动态添加**：总 Agent 下载或用户手动添加，自动或手动打标签

#### Skill（技能）
- **多个 Tool 的组合**，完成一个子任务
- 在 MVP 中暂不独立，直接体现在 Agent 配置中

#### Agent（智能体）
- **拥有特定标签的工具集**，执行特定任务
- **配置项**：名称、描述、系统提示词、工具标签列表、选用 LLM
- **可独立运行**，也可在画布中与其他 Agent 连接
- **可克隆**：从已有 Agent 快速创建新 Agent

#### Canvas（画布）
- **Agent 的编排容器**
- 支持创建、保存、加载多个画布
- 节点（Agent）只能在画布上运行
- 连线代表数据流向

#### Master Agent（总 Agent）
- **项目级控制器**，拥有全局视图
- **权限范围**：
  - ✅ 创建/删除/修改画布
  - ✅ 创建/删除/修改 Agent
  - ✅ 调整 Agent 连接关系
  - ✅ 修改 Agent 提示词、工具标签
  - ✅ 管理工具（下载、添加、打标签）
  - ❌ 不能修改应用源代码
- **记忆系统**：用户画像、偏好、现有节点、节点用法（不记无关细节）
- **UI 位置**：主界面独立面板，下方输入框，上方输出/历史

## 三、技术架构

### 3.1 技术栈

| 层级 | 技术 | 理由 |
|------|------|------|
| **桌面框架** | Tauri | 比 Electron 轻量，安装包小 |
| **前端框架** | React + Vite | 简单、快速、生态好 |
| **语言** | TypeScript | 类型安全（但 minimal types） |
| **画布组件** | React Flow | 成熟的拖拽画布库，开箱即用 |
| **UI 组件** | Ant Design | 开箱即用的组件库 |
| **状态管理** | Zustand | 比 Redux 简单 10 倍 |
| **存储** | 本地 JSON + Markdown | 简单直接，无需数据库 |
| **LLM 调用** | 直接 HTTP 请求 | 不依赖 SDK，简单可控 |
| **工具实现** | Python 脚本 | 优先三方库（如 `requests`） |
| **Python 服务** | FastAPI | 本地 HTTP 服务，供 Tauri 调用 |

### 3.2 Tauri ↔ Python 通信架构

```
┌─────────────────┐
│   Tauri (Rust)  │
│   + Web 前端    │
└────────┬────────┘
         │ HTTP 请求
         │ (localhost:18081)
         ▼
┌─────────────────┐
│  Python FastAPI │
│  (本地服务)      │
└────────┬────────┘
         │ 调用
         ▼
┌─────────────────┐
│  Python 工具脚本 │
│  (openpyxl 等)  │
└─────────────────┘
```

**通信方式**：
- Tauri 启动时自动启动 Python FastAPI 服务
- 所有工具通过 HTTP API 调用（`POST /tools/{tool_name}`）
- 优点：清晰分离、易错误处理、可扩展
- 缺点：需管理 Python 进程生命周期

**进程管理**：
- Tauri 启动时检查 Python 环境
- 自动启动 Python 服务（后台进程）
- 应用关闭时自动停止 Python 服务
- 服务异常时提示用户重启

### 3.3 Python 依赖管理

**方案**：
- 项目内包含 `requirements.txt`
- 首次运行时自动创建 `venv/` 虚拟环境并安装依赖
- 提供手动安装指南（`docs/setup.md`）
- 依赖冲突：使用独立的 virtualenv 隔离

1. **优先**：现成的 Python 三方库（如 `requests`、`openpyxl`、`PyPDF2`）

### 4.3 数据版本管理

**版本字段**：所有 JSON 配置文件必须包含 `version` 字段
```json
{
  "version": "1.0.0",
  "data": { ... }
}
```

**版本规范**：
- 格式：语义化版本（major.minor.patch）
- major：不兼容的格式变更
- minor：向后兼容的功能新增
- patch：向后兼容的问题修复

**迁移策略**：
- 应用启动时检查所有配置文件的 `version` 字段
- 如版本低于当前应用支持的版本，自动执行迁移脚本
- 迁移脚本按版本顺序执行（1.0.0 → 1.1.0 → 2.0.0）
- 迁移失败时提示用户，保留原始文件（`.backup`）

**迁移示例**：
```typescript
// migrations/v1.1.0.ts
export function migrate(data: any) {
  // 1.0.0 → 1.1.0: 添加新字段
  if (!data.newField) {
    data.newField = 'default';
  }
  data.version = '1.1.0';
  return data;
}
```

## 五、执行机制

### 5.1 执行模式
- **顺序执行**：A → B → C 有箭头连接，按依赖顺序执行
  - 前序节点失败 → 整个画布标记为失败，但**已通过节点保持绿色成功态**
- **并行执行**：依赖已满足的节点会进入并行队列；当前单画布最多同时运行 4 个节点，启动错峰 80ms
  - 单个并行节点失败只影响它自己的后续依赖，不会跳过无依赖的旁支节点
- **多画布并行**：可选择所有已打开且有节点的画布批量运行（最多 6 个画布纳入批次，当前 6 个画布并发运行）

### 5.2 启动条件校验
- Agent 执行前检查输入数据来源：
  - ✅ 上游推送（画布连线）
  - ✅ 手动输入（文件上传）
  - ✅ 历史数据（从 outputs/data/ 选择）
  - ❌ 无数据来源 → 跳过该节点

### 5.3 实时进度显示
每个 Agent 节点显示状态图标：
- 🟡 黄色：运行中（显示进度百分比或已用时间）
- 🟢 绿色打钩：成功
- 🔴 红色叉：失败
- ⚪ 灰色：未执行/已跳过

### 5.4 错误处理
- **顺序执行**：前序节点失败 → 整个画布标记为失败，但**已通过节点保持绿色成功态**
- **并行执行**：单个节点失败 → 只标记该节点失败，其他节点继续
- **画布状态**：画布整体状态显示为"失败"（红色标识），但已完成节点的状态不受影响

### 5.5 执行控制
- **取消/中止**：用户可取消正在执行的 Agent（使用 AbortController）
- **重试机制**：Python `llm-calling` 对瞬断/超时/429/5xx 自动重试；Runner 对断流、空回复、超时、429/5xx 等临时失败进行节点级补跑
- **执行超时**：单个 Agent 默认超时 5 分钟（可在 Agent 配置中调整）

## 十一、UI 设计规范（现代简约风格）

### 11.1 核心设计哲学
- **克制与留白**：界面不拥挤，强调"呼吸感"，利用大面积留白界定内容区块
- **功能可见性**：元素看起来可交互（轻微阴影/背景色），但不喧宾夺主
- **视觉层级**：通过字重、字号、颜色深浅区分信息主次
- **轻量化拟物**：扁平化设计，但材质有物理世界隐喻（柔和阴影、圆角）

### 11.2 整体布局
- **侧边栏主导航**：
  - 左侧固定宽度导航侧边栏（280px），右侧主要内容区域
  - 侧边栏背景：极浅灰色（#FAFAFA）或半透明磨砂材质
  - 选中状态：圆角矩形高亮（主色调淡色或系统蓝色）
- **统一标题栏**：
  - 标题栏与工具栏合并
  - 窗口控制按钮下方无明显分隔线
- **内容区域**：
  - 卡片式或列表式布局
  - 列表行高宽松，悬停时浅灰色背景反馈

### 11.3 视觉元素与材质
- **毛玻璃效果**（可选）：
  - 使用 CSS `backdrop-filter: blur(10px)`（现代浏览器支持）
  - 侧边栏、弹出菜单、模态框背景可带背景模糊效果
- **圆角**：
  - 基础容器/卡片：8px - 12px
  - 按钮/输入框：6px - 8px
  - 头像/图标：根据形状调整
- **阴影**：
  - 悬浮面板/弹窗带柔和扩散阴影
  - `box-shadow: 0 2px 8px rgba(0,0,0,0.1)`
- **配色**：
  - 背景色：#FFFFFF（浅色模式）
  - 强调色：#1890FF（系统蓝）或 #52C41A（系统绿）
  - 文字色：#000000DE（Primary）、#00000099（Secondary）、#00000066（Tertiary）

### 11.4 交互与动效
- **微交互**：
  - 按钮点击时轻微缩放反馈（`transform: scale(0.98)`）或高亮反馈
  - 列表项悬停时背景色渐变过渡（`transition: background-color 0.2s`）
- **平滑过渡**：
  - 页面切换/内容加载使用 Fade 或 Slide 动画
  - 时长 200ms - 300ms，缓动函数 `ease-out`
- **右键菜单**：
  - 支持右键唤出操作菜单
  - 样式：小字号、圆角、白色背景、柔和阴影

### 11.5 具体组件
- **按钮**：
  - Primary：实心背景（#1890FF），圆角，白色文字
  - Secondary：灰色背景（#F0F0F0）或仅边框
  - Tertiary：仅文字，悬停时显示下划线或背景
- **表单控件**：
  - 输入框：无边框或仅底边框，聚焦时边框变为系统蓝色
  - 下拉菜单：带小箭头，点击后弹出带阴影的列表层
- **字体**：
  - 优先使用 Segoe UI（Windows）/ 系统默认
  - 标题字重：600（Semibold）或 700（Bold）
  - 正文字重：400（Regular）

### 11.6 主题
- **MVP 阶段**：仅支持浅色模式
- **后续迭代**：根据需求添加深色模式

### 13.1 并发限制
- **同时运行画布数**：最多 6 个画布同时运行
- **画布节点数**：不做限制
- **Agent 并发**：受画布并发限制，无额外限制

- **单个 Agent 输出文件**：最大 10MB（LLM 生成内容的合理上限）

- **LLM 会话隔离**：同一 Agent 在不同画布执行时，会话必须隔离，不能串会话
- **工具复用**：优先使用现有项目/库，避免重复造轮子

- **清理资源**：
  - 停止 Python 服务
  - 保存所有画布状态
  - 写入日志

### 18.4 异常处理
- **Python 服务崩溃**：
  1. 检测到服务无响应（HTTP 请求超时或连接失败）
  2. 自动尝试重启 Python 服务（最多 3 次，间隔 2s/4s/8s）

  4. 记录详细日志（崩溃原因、重启尝试次数、错误信息）
- **LLM 调用失败**：重试 3 次（指数退避），失败后提示用户检查网络或 API Key

## 二十四、注意事项

1. **极简化开发**：尽量选用简单的技术栈或工具集，除非复杂语法能有效提升运行效率或减少代码行数
2. **避免重复造轮子**：有现成项目可参考、改造的，优先改造现有项目
3. **操作确认**：所有写操作（创建、修改、删除）需人工确认
4. **数据安全**：所有数据本地存储，不上传云端（个人使用，明文存储）
5. **可追溯**：所有执行结果保存历史记录，详细日志供 AI 排查
6. **LLM 会话隔离**：同一 Agent 在不同画布执行时，会话必须独立，不能串会话
7. **用户友好**：错误提示、加载状态等用户体验细节要到位
8. **工具复用**：优先使用 Python 三方库和 GitHub 优质项目
9. **跨平台兼容**：UI 设计使用跨平台方案，避免平台专有特性
10. **进程管理**：Tauri 负责管理 Python 服务生命周期
