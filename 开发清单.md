# 开发清单

> 当前版本：v3.02
> 更新日期：2026-07-10
> 用途：说明这个软件是什么、现在能做什么、用什么技术做的、一路是怎么做出来的、现在还剩什么。

---

## 一、一句话说明

一个跑在 Windows 上的桌面软件：你在画布上把多个 AI Agent 连成工作流，让它们分工协作、自动跑出结果（文档 / 表格 / 思维导图等）。里面还有一个总管 AI「姬子」，你用大白话跟它说话，它帮你操作画布、诊断失败、补工具。

面向的是**个人本地使用**，不是多人云端服务，所以数据都存在本地明文文件里，API Key 也由用户自己在应用里配置。

---

## 二、现在能做什么（功能全景）

**多画布工作台**
- 同时打开最多 20 个画布；同时运行的并发上限是 6（`MAX_CONCURRENT_RUNS=6`）。
- 页签可以打开、关闭、重命名、保存。
- 运行时用的是画布的只读快照，不会把运行状态写回原画布。

**Agent 库（两层模型）**
- 「定义」放在库里，「实例」是画布上的节点，两者靠快照复制、完全独立。
- 支持预置模板、创建、编辑、克隆、导出 / 导入、删除前引用检查、拖拽排序。

**Workflow 工作流引擎**
- 按依赖关系并行执行；某个节点失败只影响它的下游，不阻断无关的并行分支（失败隔离）。
- 门控节点支持 OR / AND / NOR。
- 节点级 LLM 重试；上游数据优先按 JSON 流转，schema 能匹配就匹配，匹配不上用 LLM 兜底转换。

**输出产物**
- 固定写到项目 `outputs/`，按 月份 → 日期 → 画布运行 → 节点产物 分级存放。
- 支持导出 Markdown、Word、Excel、思维导图。

**报告中心**
- 集中展示已结束运行的指标、历史产物、Token 用量统计（Token 只作参考，不作计费依据）。
- 只采集已结束的运行，运行中画布的半成品不提前进统计。

**姬子总 Agent**
- 多会话聊天，支持附件、联网搜索、记忆管理、上下文摘要。
- 能用自然语言操作画布，涉及关键动作会走多步确认，事后给行动摘要。
- 会话之间彼此隔离，一个会话的发送 / 取消不影响其他会话。

**姬子失败诊断与自愈**
- 节点失败后能诊断出「缺工具 / 缺库」，生成候选工具。
- 已注册工具可自动授权后原地重跑；安装新工具、落盘未确认的代码，必须用户确认，不会自动执行。

**工具系统**
- 内置工具 + 自定义工具注册表；工具元数据含工具名、中文描述、依赖、能力清单、实现原理和代码。
- 工具安装时服务端做 AST 顶层副作用扫描，防止装工具时就偷跑代码。

**Skill 系统**
- 人格与 Skill 已拆分；Skill 商店给人看的是中文名 / 描述，模型读的是英文说明、能力清单和使用规则。
- 按 Skill 内容自动归类到 工作流 / 工具 / 诊断 / 模型 页签（只影响管理界面，不改变模型实际读到的内容）。

**模型配置**
- 多厂商配置、通信测试、模型能力自动推断；旧配置会迁移重算能力，同时保留人工修正入口。

## 三、技术栈（真实现状）

| 层 | 用的东西 |
|----|----------|
| 桌面外壳 | Tauri 2（Rust） |
| 前端 | React 19 + Vite + TypeScript + Zustand + Ant Design v6 + @xyflow/react |
| 后端 | Python FastAPI，跑在 localhost:18081，由 Rust `python_manager.rs` 拉起并管生命周期 |
| Python | 3.12 系列，依赖见 `python/requirements.txt` |
| 数据存储 | 本地明文 JSON：`data/multi-agent-*.json`，由 Rust `storage_get/set/remove` 命令读写 |
| 运行环境 | 仅 Windows；UI 全中文；需要 WebView2 Runtime |

数据存放规则：开发时写项目顶层 `data/`，正式安装版写 exe 同级 `data/`，浏览器预览模式回落到 localStorage。

## 四、开发时间线（里程碑）

这个项目实际经历了几十个小版本，下面按大阶段精简，省掉零碎的修 bug 版本。

**阶段一 · 基础框架**
搭起 Tauri + React + 画布的骨架：多画布页签、Agent 库的两层模型（定义 vs 画布实例）、数据的本地明文 JSON 持久化。确定了「主干 + 功能节点」低耦合的写法。

**阶段二 · 工作流引擎**
把画布真正跑起来：依赖并行、失败隔离、门控节点、节点级重试、上游 JSON 优先流转与 schema/LLM 兜底转换。产物写入分级的 `outputs/`。

**阶段三 · 工具与 Python 后端**
接入 Python FastAPI 后端，由 Rust 管生命周期。建工具系统：内置工具 + 自定义工具注册表 + 安装时 AST 顶层副作用扫描。

**阶段四 · 姬子总控**
加入总管 Agent「姬子」：多会话、附件、联网搜索（多厂商失败切换）、记忆、上下文摘要、自然语言操作画布、多步确认。确立 prompt caching 依赖稳定前缀的约束。

**阶段五 · 自愈闭环**
姬子能诊断节点失败（缺工具 / 缺库）、生成候选工具、原地重跑；同时守住红线——新工具安装和未确认代码落盘必须人工确认。

**阶段六 · 打包与体验整理（v3.02，当前）**
- 打包方案：NSIS currentUser 安装（装到 `%LOCALAPPDATA%\Programs`，无 UAC、目录可写）+ 随包带可重定位的 CPython 3.12（python-dist），目标机免装 Python。
- 报告中心成形；Skill 与人格拆分、Skill 商店自动分类。
- 姬子外层状态栏精简：外面只留后台状态、当前模型、工具数量、待处理、问题数、画布诊断；任务计划、一键体检移进姬子抽屉。
- 模型能力识别会给旧配置补标签（仍是自动推断，保留人工修正）。
- 安全整改：AST 顶层副作用扫描堵住赋值型绕过（如 `_ = os.system(...)`）；工具安装放行口 `allow_top_level_side_effects` 全链路贯通；门控逻辑抽出纯函数并加自动化测试。

## 六、继续开发的铁律

- **低耦合**：新功能尽量放独立文件 / 目录，主文件只做接线。
- **运行用只读快照**，不写回原画布。
- **失败隔离**：某节点失败只波及其下游。
- **会话隔离**：一个会话状态不串到另一个。
- **姬子系统提示词是文件导入模式**，不要擅自恢复手动编辑。
- **prompt caching 靠稳定前缀**：`chat()` 的 `system` 只放人格块（每轮字节不变才命中缓存）；当前时间、记忆、搜索、摘要等每轮会变的内容放 `text`，不要塞进 `system`。
- **新增外部 HTTP 调用**要同步 Tauri plugin-http、Rust 插件注册、capabilities 白名单，不能用通配符放行。
- **安全红线**：生成的自定义工具必须人工审核后才落盘执行，不弱化 AST 顶层副作用扫描。
