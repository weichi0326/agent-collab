# 我的个人开发习惯

本文档是本项目开发时的默认工作约定。开始任务前应先阅读；如果我在当前对话中给出新要求，以最新要求为准，并及时更新本文档。

## 一、沟通方式

- 使用简单、直接的中文，先说结论，再说明必要原因。
- 面向非专业技术人员表达，避免堆砌术语、复杂流程和过长方案。
- 每次只推进当前明确目标，不擅自扩大问题或增加“顺便做”的事项。
- 有多种做法时，优先推荐最简单、最容易理解和维护的一种。
- 遇到阻塞或异常卡顿时及时停止并说明原因，不反复尝试低收益工具。
- 重视开发时间和 token 成本；人工检查可以可靠完成的事情，不强行自动化。

## 二、开发原则

- 优先完成最小可用闭环，再逐步完善，不提前建设暂时用不到的能力。
- 先理解现有代码和设计，再做局部修改，不进行无关重构。
- 保持现有优秀 UI 的布局、颜色、密度、导航和主要文案；接功能不能顺带重做界面。
- 功能要真实可用，不只停留在演示 UI；但每一步都应控制范围。
- 代码和目录应让后续开发者容易找到、容易理解，不为了“架构完整”增加不必要的层级。

## 三、master 与专业包

- `master` 是基础工作台，负责通用多 Agent 画布、节点编排、运行、历史和产物管理。
- 小说家、测试工程师等属于专业包；专业包基于 `master`，不能发展成与画布无关的独立应用。
- 专业包的核心价值是把专业数据、专业操作和专业工作流接到 `master` 的节点能力上。
- 专业任务应先创建一个可见、可编辑、可保存的画布，由用户确认后再运行。
- 可以同时安装多个专业包，每个专业包都必须提供与其职业场景对应的功能和画布模板。
- 第一版专业包按可信本地包处理，支持 JavaScript 和 Python，不允许包内增加 Rust；安装后重启生效。
- 专业包开发成熟后，应能从分支中剥离为可导入的独立包；干净的 `master` 导入后，应恢复对应专业功能。

## 四、分支与提交

- `master` 只保留通用基础能力；小说家功能在 `fictionist` 分支开发。
- 修改应小步提交，一个提交只解决一个清晰问题。
- 不混入无关文件、临时输出、截图、缓存或他人的未提交修改。
- 未经明确要求，不推送远端、不合并分支、不执行破坏性 Git 操作。

## 五、验证要求

- 验证力度与改动风险相匹配，只运行能证明当前修改正确的必要检查。
- 代码改动优先运行相关测试；共享功能或提交前再运行完整测试、Lint 和构建。
- UI 仍处于雏形阶段时，以人工检查常见窗口为主，不默认维护大量自动截图。
- 桌面人工检查使用 `1280x720`、`1600x900` 和 `1920x1080`。
- 没有实际运行验证命令前，不声称功能已经完成或通过。

## 六、默认执行顺序

1. 明确当前唯一目标和不做什么。
2. 阅读直接相关的代码与文档。
3. 选择最小、最直接的实现方式。
4. 完成实现，不顺带扩展范围。
5. 运行必要验证，发现问题就如实说明。
6. 简短汇报结果、改动文件和下一步。

## 七、优先级

出现冲突时，按以下顺序处理：

1. 我在当前对话中的最新明确要求。
2. 本文档记录的个人开发习惯。
3. 项目已有开发规范和架构约束。
4. 工具或开发者自行推测的偏好。
