## 三、本轮新发现的问题（v4）

### V4-S1. SSRF 防护仍未完全收紧（HIGH）— 已修复 ✅

**位置**：`llm_calling.py:104-105`

```python
    # 外部公网地址：允许（兼容用户自部署的公网 LLM 服务）
    return base_url.rstrip("/")
```

**问题**：v3 报告 V3-S2 指出删除公网放行分支，但实际代码第 104-105 行仍存在公网放行逻辑。攻击者可构造 `base_url` 指向任意公网服务器，让 Python 服务代理发送包含 `api_key` 的请求到攻击者服务器。这是当前最高安全优先级。

**影响**：API Key 通过 HTTP 请求泄露到攻击者控制的服务器。

**修复结果**：已删除公网默认放行分支；当前仅允许 `_ALLOWED_DOMAINS` 中的可信域名，非白名单公网域名会被拒绝代理请求，避免 API Key 被转发到攻击者服务器。已补充回归测试。

### V4-S2. UAC 提权 PowerShell 脚本路径注入（HIGH）— 已修复 ✅

**位置**：`file_write.py:112-143`

`_run_elevated_permission_repair` 生成临时 `.cmd` 脚本并通过 `Start-Process -Verb RunAs` 提权执行 `icacls`。`identity` 来自 `whoami /user` 命令输出（可被环境变量伪造）；`output_root` 由前端传入，`allow_outside_roots=True` 时绕过沙箱；临时脚本路径通过 `_ps_single_quote` 拼接 PowerShell 命令，若路径含特殊字符可能注入。

**影响**：精心构造的路径可能绕过保护，以管理员权限修改任意目录 ACL。

**修复结果**：已移除 `_run_elevated_permission_repair` 及 PowerShell `Start-Process -Verb RunAs` 提权路径。文件写入工具仅保留普通权限修复尝试；失败时提示用户手动授权或改用项目 `outputs/` 目录。

### V4-B3. llm_calling.py 同步阻塞事件循环（MEDIUM）— 已缓解 ✅

**位置**：`llm_calling.py:124-170`

`execute` 是 `async def`，但内部 `_post_json` 使用同步 `requests.post`，阻塞 FastAPI asyncio 事件循环。多并发请求时所有请求串行化。

**影响**：并发场景下性能严重退化。

**修复结果**：已用 `asyncio.to_thread` 包装 OpenAI/Anthropic 兼容调用，避免同步 `requests.post` 长请求直接阻塞 FastAPI 事件循环；暂不引入 `httpx` 新依赖。

### V4-B4. recomputeDerived 全量重算（MEDIUM）

**位置**：`canvasStore.ts:248-301`
