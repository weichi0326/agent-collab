# 姬子智力系统评审

评审日期：2026-07-10

评审口径：只看实际代码、测试和配置，不采信项目根目录里的开发清单、交接文档、路线图等可能过期文本。

评分口径：把 Codex 级别的工程智能体视为 10 分，1 分作为及格线。

## 总分

姬子当前智力评分：2.4 / 10，已经及格。

一句大白话：姬子不是普通聊天框了，它已经像一个“会看项目、会分诊、会记一点偏好、会搜索、会按固定动作改画布和装工具的项目助理”。但它还不是 Codex 这种能自己反复观察、规划、执行、验证、重规划的工程智能体。

强模型加持下，姬子的体感大约能到 3 / 10。换成普通模型，可能只有 1.5 到 2 / 10。

## 现在已经有的脑子

姬子不是纯靠一句 prompt 在硬聊。代码里已经有一套回合规划器，会把用户请求拆成聊天、操作、生成工具、询问选择、系统体检等不同类型。

它能操作画布里的固定动作，例如创建和重命名画布、创建 Agent 或节点、连接节点、删除节点、调整输出格式、运行画布。这说明它已经具备“把话变成产品内动作”的能力。

它接入了 Skill 自动选择、语义记忆筛选、是否需要搜索的判断、搜索结果筛选、对话摘要、长期记忆、图片上下文和来源展示。也就是说，它不只是回答当前一句话，而是开始有一点“上下文感”和“工具感”。

它还能生成 Python 工具，并走人工确认、安装、烟雾测试、最多两轮修复的流程。节点缺工具标签时，也有自动补标签并重跑的自愈逻辑。这部分已经接近“小型自动化助手”的范畴。

## 最大短板

姬子现在最大的短板，是缺少通用的自主闭环。

更直白地说：它现在像一个会做很多固定菜单动作的聪明助手，但还不太会自己连续办事。它能计划一次，也能执行一些动作；可是一旦中间出错，它没有稳定的“看结果、判断哪里坏了、修正方案、再试一次、最后验收”的通用循环。

所以它适合做明确、短链路、产品内已有动作比较清楚的任务。不适合处理模糊、多步、需要持续试错和自我校准的工程任务。

## 主要问题

1. 规划基本是一次性的

   回合规划器会产出行动计划，但失败后很多路径会回到聊天或局部兜底，没有统一的自主循环。它不像 Codex 那样天然围绕“观察、计划、执行、验证、重规划”跑。

2. 运行上下文太薄

   当前给姬子的上下文主要是画布名、节点名、Agent 简介、工具名等摘要信息。它看不到足够完整的结构化画布状态，例如连线、节点配置、输入输出、运行结果和失败细节。一个智能体看不清现场，就很难聪明地诊断现场。

3. 自愈容易误诊

   当运行诊断解析失败时，逻辑偏向默认认为缺工具，进而尝试生成工具。这对“真缺工具”的场景有帮助，但对配置错误、输入错误、模型输出错误等场景容易开错药。

4. 长期记忆还比较原始

   记忆目前更像追加式笔记，只做完全相同文本的去重。它缺少替换、过期、置信度、来源、冲突处理。时间久了以后，记忆可能会变成一堆互相打架的小纸条。

5. 多步动作缺少事务保护

   多个动作顺序执行时，如果做到一半失败，系统可能留下半成品状态。比如已经创建了一部分节点，但后续连接或配置失败。它目前没有足够稳的预检、快照和回滚。

6. 搜索能力偏浅

   搜索质量判断主要吃标题和 snippet，没有真正读取网页正文，也没有稳定的交叉验证机制。它能“找线索”，但不算会“做研究”。

7. 默认提示词与实际能力有冲突

   默认系统提示词里仍有“姬子不能直接修改画布”之类的旧表述，但实际代码已经支持画布操作。这会让模型对自己能不能动手产生混乱。

8. 测试覆盖还没测到真正的智力闭环

   现有测试主要覆盖 JSON 解析器、reducer 等基础逻辑。完整智能回合、失败恢复、误诊、记忆冲突、工具结果语义判断等关键场景还没有形成系统测试。

## 提升方案

第一步，先做真实任务评测集。

建议收集 50 到 100 个真实用户任务，覆盖画布编辑、Agent 配置、工具安装、搜索问答、失败修复、记忆使用、多轮澄清等场景。没有评测集，智力提升会很容易变成凭感觉调 prompt。

第二步，实现有限步自主循环。

给姬子加一个明确的循环：观察当前状态，制定下一步，只执行一小步，读取结果，判断是否成功，再继续或停止。循环必须有最大步数、失败次数、用户确认门槛和清晰的终止理由。

第三步，补足结构化上下文。

让姬子看到完整画布结构、节点配置、连线、输入输出、最近运行结果、错误栈和工具返回值。不要只给它一个概述，要让它能看到足够多的现场证据。

第四步，给动作系统加预检和回滚。

执行多步动作前先检查条件是否满足。执行前保存必要快照。失败后能恢复到执行前状态，或者至少给出可读的半成品状态和人工修复建议。

第五步，收紧自愈诊断。

解析失败时不要默认判定缺工具。应该先区分：缺工具、工具参数错、节点配置错、输入缺失、模型输出不合格、网络或后端错误。只有证据足够时才进入工具生成。

第六步，升级长期记忆。

记忆需要有来源、时间、置信度、适用范围和冲突处理。新记忆进入时，要能覆盖旧记忆、标记过期信息，或者在冲突时要求用户确认。

第七步，补端到端智力测试。

重点测试完整回合，而不是只测工具函数。应该加入：计划成功、计划部分失败、误诊避免、工具自愈、记忆冲突、搜索不足、用户中途改口、画布半失败恢复等场景。

## 最后判断

姬子现在已经过了“只是一个聊天窗口”的阶段，进入了“产品内智能助理”的早期阶段。它的地基是有的，尤其是动作系统、工具生成、自愈入口和上下文管线都已经搭起来了。

但它真正缺的是一套稳定的执行大脑：能持续看现场、分解任务、执行一步、验收一步、失败后重来。这个补上以后，姬子的智力会从“会帮忙”明显跨到“能办事”。
