# 小说家与工作台集成规划

> 状态：待实施  
> 制定日期：2026-07-23  
> 适用分支：`fictionist`  
> 核心目标：让小说家使用现有画布和 Agent 运行能力完成真实写作任务，同时保持专业数据、专业界面和基础工作台相互隔离。

## 一、结论

小说家不应成为另一套 Agent 执行器，也不应把章节、设定等数据塞进 `canvasStore`。

正确的职责关系是：

```text
小说家：管理作品、章节、设定和时间线
                 |
                 | 创建任务、提供上下文、接收草稿
                 v
专业任务桥接层：记录任务来源、转换输入、识别输出
                 |
                 v
工作台：编辑和运行 Agent 画布，保存运行历史与产物
```

第一阶段只完成一条真实闭环：

```text
选择已有章节
  -> 点击“续写下一章”
  -> 创建小说续写画布
  -> 切换到工作台并运行
  -> 得到明确标记的章节草稿
  -> 用户预览和确认
  -> 保存为下一章
```

先证明这条链路可靠，再扩展润色、审校、设定提取和时间线同步。

## 二、当前项目事实

### 2.1 已具备的能力

- 已有独立的小说家一级界面和书架、正文、设定库、时间线、工作流二级界面。
- `canvasStore` 已支持创建、导入、保存、打开和运行画布。
- `agentRunner` 已支持画布拓扑运行、并行调度、中止、重跑和产物保存。
- 节点结果会写入 Markdown、Word、Excel 或思维导图文件，并生成对应的 `data.json`。
- 工作台已有运行历史、只读运行副本和产物删除标记。
- 项目已有基于 Tauri 的本地 JSON 持久化能力，并在浏览器开发环境回退到 `localStorage`。

### 2.2 尚未打通的部分

- 小说作品和章节仍是 `FictionistWorkspace.tsx` 内的演示状态，重启后丢失。
- “续写下一章”目前只显示提示，不会创建画布。
- `RunCanvasResult` 只返回节点数和写出文件数，调用方无法可靠确认哪个节点是章节正文。
- `Canvas` 没有记录专业包、作品、章节和任务来源。
- 小说家入口仍由 `App.tsx` 和 `TitleBar.tsx` 直接写死，不是真正的专业包注册结果。
- 当前存储命令单个值上限为 10 MB，不适合把整部长篇小说作为一个 Zustand JSON 反复整体写入。

## 三、实施原则

1. **保持两套界面独立**：小说家继续管理内容，工作台继续编辑流程，不把画布硬塞进小说编辑页。
2. **复用执行能力**：小说家不直接调用模型和工具，只创建专业任务并交给现有 `agentRunner`。
3. **通过门面通信**：小说组件不能直接深层调用 `canvasStore` 或 `agentRunner` 内部实现。
4. **明确结果身份**：通过稳定的输出角色识别章节草稿，禁止依赖节点名称或文件名猜测。
5. **人工确认回写**：运行成功只产生候选草稿，用户确认后才创建或覆盖章节。
6. **专业数据独立**：作品数据与画布数据分别持久化，卸载小说包不能破坏基础工作台。
7. **默认保留用户数据**：卸载专业包时默认保留作品，删除作品必须单独说明并二次确认。
8. **逐步抽象**：先完成小说续写闭环，再提炼通用专业包能力，避免一开始构建过大的插件平台。
9. **保持兼容**：现有普通画布、运行历史、报告中心和设置中心行为不能改变。
10. **桌面优先**：只验收 Windows 桌面布局，不为手机和平板增加额外分支。

## 四、目标架构

### 4.1 专业包注册层

新增轻量级专业包契约，初期只注册应用内已经受信任的模块：

```ts
interface ProfessionalPackageManifest {
  id: string;
  name: string;
  version: string;
  apiVersion: string;
  workspace: {
    viewId: string;
    label: string;
  };
  taskTemplates: ProfessionalTaskTemplate[];
  ownedStoragePrefixes: string[];
  capabilities: string[];
}
```

小说家清单至少声明：

- 包 ID：`fictionist`
- 工作区入口：`小说家`
- 首个任务模板：`continue-chapter`
- 后续任务模板：润色、审校、设定提取、线索检查
- 所属存储前缀：`multi-agent-fictionist-*`
- 卸载后会失去的能力说明

第一阶段的“安装/卸载”是受控注册，不允许运行用户导入的任意 JavaScript。真正支持外部代码包需要签名、权限、隔离和版本兼容机制，另行规划。

### 4.2 小说领域层

小说领域只管理小说数据，不管理画布节点：

```text
fictionist/
  types.ts                 作品、卷、章节、设定、时间线类型
  store.ts                 当前作品、选择状态和元数据
  chapterRepository.ts     章节正文分文件读取与原子保存
  selectors.ts             字数、章节排序、上下文筛选
  contextBuilder.ts        续写上下文构造
  validation.ts            标题、顺序、引用完整性校验
```

建议的核心数据：

```ts
interface FictionProject {
  id: string;
  title: string;
  genre: string;
  status: 'drafting' | 'paused' | 'archived';
  volumeIds: string[];
  createdAt: string;
  updatedAt: string;
}

interface FictionVolume {
  id: string;
  projectId: string;
  title: string;
  order: number;
  chapterIds: string[];
}

interface FictionChapter {
  id: string;
  projectId: string;
  volumeId: string;
  title: string;
  order: number;
  status: 'outline' | 'draft' | 'revised' | 'final';
  wordCount: number;
  contentStorageKey: string;
  revision: number;
  sourceTaskId?: string;
  createdAt: string;
  updatedAt: string;
}
```

章节正文不直接放进一个总 Zustand 状态。每章使用单独、安全的 ASCII 存储键，例如：

```text
multi-agent-fictionist-chapter-<chapterId>
```

这样可以避免长篇小说逼近单文件 10 MB 上限，也能减少每次输入一个字就重写整部作品的问题。元数据仍可保存在 `multi-agent-fictionist-index` 中。

### 4.3 专业任务桥接层

桥接层是小说家和工作台唯一允许的连接点：

```text
professionalTasks/
  types.ts                 通用任务关联和输出角色
  taskStore.ts             待运行、运行中、待确认、已接收状态
  canvasBridge.ts          创建任务画布、切换画布、关联运行结果

fictionist/
  canvasBridge.ts          小说上下文与通用任务之间的转换
  templates/
    continueChapter.ts     续写下一章画布模板
```

桥接层负责：

- 从小说项目中读取所需上下文。
- 克隆专业包提供的画布模板。
- 为画布附加通用任务来源信息。
- 切换到工作台并激活新画布。
- 接收运行结果并找到指定输出角色。
- 将结果登记为“待确认草稿”。
- 用户确认后调用小说仓储保存下一章。

桥接层不负责：

- 直接执行模型请求。
- 修改普通画布业务规则。
- 自动覆盖任何已有章节。
- 删除用户作品。

### 4.4 画布任务来源

在 `Canvas` 上增加可选且通用的任务来源，不增加小说专属字段：

```ts
interface ProfessionalTaskOrigin {
  packageId: string;
  taskId: string;
  taskType: string;
  sourceEntityIds: string[];
  returnView: string;
}

interface Canvas {
  // 现有字段保持不变
  taskOrigin?: ProfessionalTaskOrigin;
}
```

普通画布没有 `taskOrigin`，行为完全不变。专业任务画布可以在工作台顶部显示：

```text
来自《七号泊位》 · 续写第 5 章    [返回作品] [查看任务上下文]
```

### 4.5 稳定的输出角色

节点需要一个通用、可选的结果角色，例如：

```ts
interface AgentNodeData {
  resultRole?: string;
}
```

小说续写模板的最终节点声明：

```text
resultRole = "fictionist.chapter-draft"
```

运行结果扩展为：

```ts
interface RunOutputRef {
  nodeId: string;
  resultRole?: string;
  label: string;
  summary?: string;
  path?: string;
  dataPath?: string;
}

interface RunCanvasResult {
  runId: string;
  runCanvasId: string;
  nodeCount: number;
  writtenCount: number;
  outputs: RunOutputRef[];
}
```

桥接层只接受 `fictionist.chapter-draft`。找不到、找到多个或格式错误时，应将任务标记为失败并要求用户检查画布，不能静默选择第一个文件。

## 五、首个真实用户流程

### 5.1 准备续写

用户在正文页点击“续写下一章”后，弹窗展示实际将注入的内容：

- 当前章完整正文。
- 当前章结尾摘要。
- 用户勾选的人物设定。
- 用户勾选的世界设定。
- 尚未回收的线索。
- 下一章目标、预计字数和写作要求。

用户可以取消某项上下文，避免一次性把整本设定库塞入模型。

### 5.2 创建画布

确认后执行：

1. 保存当前章节未保存的修改；保存失败则停止。
2. 创建 `ProfessionalTask` 记录。
3. 克隆 `continue-chapter` 模板并生成新节点 ID。
4. 把上下文写入专用输入节点或任务上下文引用。
5. 给最终节点设置 `fictionist.chapter-draft`。
6. 给画布写入 `taskOrigin`。
7. 切换到工作台并激活该画布。

建议模板最初只保留三个节点：

```text
[章节上下文] -> [续写作者] -> [一致性检查与定稿]
```

节点越少，第一阶段越容易判断错误来自数据、模型还是运行器。

### 5.3 运行和返回

- 用户仍使用工作台现有“运行”按钮。
- 工作台正常显示各节点状态和产物。
- 运行成功后，专业任务桥接层登记最终草稿。
- 顶部出现“草稿已生成，可送回小说家”。
- 用户可以先打开产物查看，也可以返回小说家预览。

### 5.4 确认保存

返回小说家后显示待确认草稿，并提供：

- 修改草稿。
- 放弃本次草稿。
- 保存为下一章。
- 重新打开对应画布。

“保存为下一章”必须：

1. 自动建议章序和标题，但允许修改。
2. 检查目标章序是否冲突。
3. 单独写入章节正文。
4. 成功后更新作品元数据和字数。
5. 最后将任务标记为 `accepted`。

如果正文写入成功但元数据更新失败，要保留可恢复记录，不能让正文变成无法找到的孤立文件。

## 六、状态与生命周期

专业任务状态建议为：

```text
preparing -> ready -> running -> review_required -> accepted
                         |              |
                         v              v
                       failed        discarded
```

关键规则：

- 应用意外退出后，`preparing` 和 `running` 任务要恢复为可理解的中断状态。
- 删除任务画布前，如果存在尚未接收的草稿，要先提醒用户。
- 删除运行产物后，任务记录应同步标记结果不可用。
- 接收章节后仍保留 `taskId`、`canvasId`、`runId`，方便追溯来源。
- 删除小说专业包时，不得删除普通画布和普通 Agent。

## 七、界面结合方式

不改变现有一级胶囊切换结构，只增加必要的上下文提示。

### 小说家侧

- 正文页保留“续写下一章”。
- 增加“待确认草稿”状态和预览区。
- 工作流页展示模板及其最近运行状态。
- 任务存在时可点击“在工作台中打开”。

### 工作台侧

- 专业任务画布顶部增加一条紧凑任务条。
- 提供“返回作品”和“查看上下文”。
- 运行完成后提供“送回小说家”。
- 普通画布不显示任务条，也不改变现有布局。

### 导航保护

- 小说章节有未保存修改时，离开作品前提示。
- 专业画布正在运行时，返回小说家不应中止任务。
- 回到小说家后仍能查看运行进度或重新进入画布。

## 八、专业包安装与卸载边界

### 第一阶段

小说家模块仍随 `fictionist` 分支构建，但由注册表决定是否显示入口。这样可以先验证边界和卸载行为，不同时引入外部代码执行风险。

### 后续数据包导入

可导入的安全专业包先限制为声明式内容：

- 清单和版本信息。
- 画布模板。
- Agent 模板和提示词。
- 数据 schema。
- 图标和静态资源。
- 能力与清理说明。

它不能直接执行任意前端 JavaScript、Rust 或 Python 代码。

### 真正的外部界面包

如果以后要求导入后出现完全新的界面和交互，需要单独建设插件宿主，包括：

- 包签名和来源校验。
- API 版本兼容。
- 权限声明与用户授权。
- 独立 WebView 或其他隔离环境。
- 文件、网络、模型和工具访问边界。
- 崩溃隔离和升级迁移。

这不属于小说续写首个闭环，不能与当前改造混在一次提交中。

### 卸载行为

卸载小说包时分两步：

1. 移除小说家入口、模板和包管理的专属资源。
2. 询问作品数据如何处理：
   - 默认：保留本地作品数据，重新安装后恢复。
   - 可选：导出后删除。
   - 高风险：直接删除，必须列出丧失的能力和数据并二次确认。

正在运行的专业任务存在时禁止卸载。

## 十、分阶段实施计划

### 阶段 0：保护当前 UI 基线

目标：先固定已经认可的界面，避免功能接入时无法区分视觉变化和逻辑回归。

- 为当前书架、正文、设定库、时间线和工作流补齐关键渲染测试。
- 保留当前桌面布局，不做额外视觉重构。
- 确认当前未提交 UI 改动独立成中文提交。

验收：全量前端测试、Lint、生产构建通过。

### 阶段 1：真实作品与章节存储

目标：新建作品、创建章节、编辑、保存和重启恢复全部真实可用。

- 新增小说领域类型、元数据 store 和章节仓储。
- 把演示数据变为首次使用时可选的示例数据或测试 fixture。
- 新建作品和章节改为真实持久化。
- 实现未保存状态、保存失败提示和字数重算。
- 将小说存储键纳入数据清理说明和清理逻辑。

验收：重启应用后作品和章节仍存在；保存失败时 UI 不得显示成功。

### 阶段 2：通用专业任务和续写模板

目标：从小说家创建一张有来源信息的真实工作台画布。

- 新增专业任务类型、任务 store 和桥接门面。
- 增加通用 `taskOrigin`。
- 增加可以返回新画布 ID 的模板创建接口。
- 建立三节点续写模板。
- 实现“续写下一章”确认后切换工作台。
- 在工作台显示任务来源和“返回作品”。

验收：普通画布行为不变；专业画布能够追溯到正确作品和源章节。

### 阶段 3：运行结果回传

目标：运行结束后得到唯一、可验证、可回写的章节草稿。

- 增加节点 `resultRole`。
- 扩展 `RunCanvasResult` 和运行器结果收集。
- 将运行结果与 `ProfessionalTask` 关联。
- 在小说家显示待确认草稿。
- 实现修改、放弃、保存为下一章。
- 处理产物缺失、多个结果和任务中止。

验收：运行成功不会自动覆盖章节；只有用户确认后才出现新章节。

### 阶段 4：小说工作流管理

目标：让小说工作流页成为真实模板管理入口。

- 查看包内模板。
- 从模板创建或打开画布。
- 允许用户复制模板后自定义。
- 清楚区分“包内模板”和“用户副本”。
- 包升级不得覆盖用户副本。

验收：用户修改自己的画布不会污染专业包原始模板。

### 阶段 5：设定与时间线参与工作流

目标：让专业数据参与上下文，而不是继续作为静态页面。

- 设定条目真实增删改查。
- 时间线事件真实增删改查。
- 建立章节、人物、地点、事件的引用关系。
- 续写时按引用和用户选择注入上下文。
- 新增“从章节提取候选设定”，仍需用户确认后写入正式设定库。

验收：模型输出不能直接修改正式设定；所有自动提取结果先进入候选区。

### 阶段 6：安装、禁用与卸载

目标：验证小说家确实遵守专业包生命周期。

- 增加包安装状态。
- 未安装时不显示小说家入口。
- 禁用只隐藏能力，不删除数据。
- 卸载移除包资源并明确提示丧失的能力。
- 作品保留、导出后删除、直接删除分别测试。

验收：卸载后基础工作台、普通 Agent、普通画布和运行历史仍可正常使用。

## 十一、建议提交拆分

每个提交保持可运行，中文提交日志建议如下：

1. `测试：固定小说家界面与导航基线`
2. `功能：增加小说作品与章节持久化`
3. `功能：增加专业任务来源与续写画布模板`
4. `功能：打通画布运行结果与章节草稿回传`
5. `功能：增加章节草稿确认与保存流程`
6. `功能：完善小说专业包禁用与卸载处理`

不要把持久化、运行器返回值、UI 大拆分和完整设定库一次性放进同一个提交。

## 十二、测试要求

### 单元测试

- 小说项目、卷和章节排序。
- 字数统计。
- 存储键生成和非法 ID 拒绝。

## 十三、主要风险与处理

### 风险 1：运行结果识别错误

处理：使用唯一 `resultRole`，禁止根据节点名称、产物顺序或文件名判断。

### 风险 2：章节被自动覆盖

处理：所有结果先进入待确认草稿；覆盖已有章节必须再次确认并生成修订记录。

### 风险 3：长篇小说导致持久化过大

处理：章节正文独立存储，元数据与正文分离，避免整本书重复序列化。

### 风险 4：小说组件和画布 store 双向耦合

处理：只允许通过 `fictionist/index.ts` 和 `professionalTasks/canvasBridge.ts` 门面调用。

### 风险 5：卸载误删基础资源

处理：所有包资源必须带 `ownerPackageId`；没有归属或归属为用户的资源不得由卸载器删除。

### 风险 6：外部包成为任意代码执行入口

处理：第一版只支持声明式包；可执行插件必须等权限、签名和隔离方案成熟后再开放。

### 风险 7：工作台运行流程被小说逻辑污染

处理：`agentRunner` 只返回通用结果，不导入任何 fictionist 模块；专业任务由上层桥接层消费。

## 十四、首个闭环完成标准

同时满足以下条件，才算“小说家已经开始和工作台结合”：

- 用户创建的作品和章节重启后仍存在。
- “续写下一章”能创建并打开真实工作台画布。
- 画布明确显示来自哪部作品和哪个章节。
- 用户可以修改节点并正常运行现有 Agent 流程。
- 系统只接收带 `fictionist.chapter-draft` 角色的结果。
- 草稿可以返回小说家预览和修改。
- 用户确认后才保存为下一章。
- 运行失败、中止或产物缺失不会生成空章节。
- 普通工作台功能和普通画布数据不受影响。
- 小说包被禁用后，基础应用仍保持干净可用。

## 十五、暂不实施的内容

以下内容不进入首个闭环：

- 多人协作和云同步。
- Git 式复杂分支合并。
- 自动改写正式设定库。
- 全书向量数据库和语义检索。
- 任意第三方 JavaScript/Python/Rust 插件执行。
- 专业包在线市场和自动更新。
- 手机和平板布局。
- 一次性实现所有小说工作流。

这些能力应在首个续写闭环稳定后分别立项，避免扩大当前改动范围。
