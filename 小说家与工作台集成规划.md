# 小说家与工作台集成规划

> 状态：待实施
>
> 修订日期：2026-07-23
>
> 适用分支：`fictionist`
>
> 上位规范：[docs/专业包开发规范.md](docs/专业包开发规范.md)
>
> 核心目标：把现有小说家 UI 接入真实数据和 master 多 Agent 画布，并最终提取成可由干净 master 导入的小说家专业包。

## 一、项目定位

小说家是专业包体系的第一个验证项目，不是 master 的永久内置业务。

```text
小说家专业包
  负责作品、章节、设定、写作任务和草稿确认
                    |
                    | 创建专业任务、整理上下文、接收结果
                    v
master 工作台
  负责画布、Agent 节点、模型、工具、运行历史和产物
```

小说家不能建立自己的 Agent 运行器。所有需要 AI 协作的写作任务都必须创建并打开真实 master 画布，由用户检查、修改和运行。

开发完成后，小说家专属代码和资源应能打包为 `.mapp`。干净 master 导入该包并重启后，应得到与 `fictionist` 分支一致的小说家功能。

## 二、第一阶段范围

第一阶段只打通一条完整、可恢复、可验证的用户流程：

```text
新建作品
  -> 新建和编辑章节
  -> 保存并在重启后恢复
  -> 点击“续写下一章”
  -> 预览和选择写作上下文
  -> 创建一张可见、可编辑的续写画布
  -> 用户在 master 工作台点击运行
  -> 草稿返回小说家
  -> 用户修改并确认
  -> 保存为下一章
```

第一阶段暂不把设定库和时间线改成真实数据，它们继续保持现有演示状态。先验证小说家与 master 画布的核心关系，再扩展其他写作能力。

## 三、当前项目事实

### 3.1 已有能力

- 已有小说家一级入口和书架、正文、设定库、时间线、工作流界面。
- 当前视觉设计已被认可，作为后续功能接入的 UI 基线。
- `canvasStore` 已支持画布创建、保存、打开、运行副本和运行历史。
- `agentRunner` 已支持拓扑运行、并发、中止、重跑、产物保存和失败状态。
- 节点可以输出 Markdown、Word、Excel、思维导图和对应 `data.json`。
- Tauri 已提供本地 JSON 原子存储，浏览器开发环境回退到 `localStorage`。

### 3.2 需要补齐

- 作品和章节仍是 `FictionistWorkspace.tsx` 内的演示状态。
- 切换离开小说家时组件会卸载，未保存正文可能丢失。
- “续写下一章”不会创建真实任务和画布。
- 当前运行返回值不能说明哪个输出是章节草稿。
- 运行完成结果只由运行按钮消费，专业任务没有统一接收入口。
- 画布、已保存画布、运行副本和运行历史没有完整的专业任务来源。
- 小说家入口仍由 `App.tsx` 和 `TitleBar.tsx` 写死。
- 桌面数据清理按固定文件白名单工作，不会自动清理动态章节键。
- 当前代码还不能把小说家提取为可导入 `.mapp`。

## 四、不可改变的实施原则

1. **保留现有 UI**：只增加必要的真实状态，不做无关视觉重构。
2. **专业数据独立**：作品和章节不进入 `canvasStore`。
3. **画布是唯一执行入口**：小说家不直接调用模型完成续写。
4. **画布必须可见可编辑**：第一版不做隐藏画布的一键运行。
5. **通过稳定门面通信**：小说家不得深层导入 master store 或运行器。
6. **结果身份稳定**：只接收带指定输出角色的草稿。
7. **人工确认回写**：运行成功只产生候选草稿。
8. **失败不能伪装成功**：保存、运行、结果读取失败必须保留可恢复状态。
9. **普通画布不受影响**：没有专业任务来源的画布行为保持不变。
10. **最终可提取**：分支内开发路径与 `.mapp` 导入路径不能形成两套实现。
11. **多个专业包可共存**：小说家的命名、存储和工具不能占用全局通用名称。
12. **Windows 桌面优先**：只验收桌面布局，不扩展手机和平板分支。

## 五、代码与包边界

### 5.1 master 通用层

master 新增或完善通用能力：

```text
professionalHost/
  包工作区注册
  专业任务记录
  专业任务与画布关联
  画布运行生命周期事件
  专业输出收集
  包命名空间存储
```

这些模块不能导入任何 `fictionist` 类型。

### 5.2 小说家专业包层

小说家负责：

```text
作品与章节数据
现有小说家界面
续写上下文构造
续写画布模板
章节草稿识别和预览
用户确认后的章节保存
小说数据迁移、导出和清理说明
```

分支开发阶段可以由 master 的内置开发加载器加载小说家源码，但必须走与未来导入包相同的注册、任务、画布和存储门面。

### 5.3 最终专业包

小说家最终打包为：

```text
fictionist-<version>.mapp
  manifest.json
  frontend/index.js
  assets/
  migrations/
  schemas/
  python/tools/       可选
```

第一版包代码由用户完全信任，安装后重启生效。包可以包含前端 JavaScript 和 Python 工具，不包含 Rust 扩展。

## 六、小说数据设计

### 6.1 元数据索引

使用带版本的小说索引保存轻量元数据：

```ts
interface FictionistIndex {
  schemaVersion: 1;
  projects: Record<string, FictionProject>;
  volumes: Record<string, FictionVolume>;
  chapters: Record<string, FictionChapter>;
  updatedAt: string;
}
```

核心实体至少包含：

```ts
interface FictionProject {
  id: string;
  title: string;
  genre: string;
  status: 'drafting' | 'paused' | 'archived';
  volumeIds: string[];
  createdAt: string;
  updatedAt: string;
}

interface FictionVolume {
  id: string;
  projectId: string;
  title: string;
  chapterIds: string[];
}

interface FictionChapter {
  id: string;
  projectId: string;
  volumeId: string;
  title: string;
  status: 'outline' | 'draft' | 'revised' | 'final';
  wordCount: number;
  revision: number;
  sourceTaskId?: string;
  createdAt: string;
  updatedAt: string;
}
```

卷中的 `chapterIds` 是章序事实源，不再同时保存独立 `order` 字段。章节正文存储键由仓储根据 `chapterId` 推导，不写进实体。

### 6.2 正文分开保存

```text
multi-agent-fictionist-index
multi-agent-fictionist-chapter-<chapterId>
```

不能把整部长篇小说放进一个 Zustand JSON。每章单独保存可以避免逼近单值 10 MB 上限，也避免编辑一个字就重写整本书。

所有 ID 使用项目统一的安全 ASCII ID 生成器。存储仓储负责校验 ID、限制正文大小和计算字数。

### 6.3 编辑和导航保护

- 当前正文和未保存状态放在小说领域 store，不留在会随页面卸载的局部组件状态中。
- 所有一级页面切换走统一导航门面。
- 离开小说家前先尝试保存；保存失败时阻止离开。
- 用户明确选择放弃修改时才丢弃内存草稿。
- 应用关闭和专业包禁用同样执行未保存检查。

### 6.4 原子接收草稿

接收草稿需要小型恢复记录：

```text
写入待执行变更
  -> 写入新章节正文
  -> 更新小说索引
  -> 将专业任务标记为已接收
  -> 清除恢复记录
```

启动时检查未完成变更。如果正文已写入但索引更新失败，应完成恢复或明确提示，不能留下无法找到的章节文件。

## 七、专业任务与画布

### 7.1 专业任务记录

通用专业任务至少记录：

```ts
interface ProfessionalTask {
  id: string;
  packageId: string;
  taskType: string;
  status: 'preparing' | 'ready' | 'running' | 'review_required'
    | 'accepted' | 'failed' | 'discarded' | 'interrupted';
  sourceRefs: Array<{ type: string; id: string; revision?: number }>;
  canvasId?: string;
  runId?: string;
  runCanvasId?: string;
  outputRefs: RunOutputRef[];
  createdAt: string;
  updatedAt: string;
}
```

小说专属的下一章目标、章节 ID 和返回位置放在小说家任务载荷中，不塞进通用画布类型。

### 7.2 画布来源

画布只保存最小通用来源：

```ts
interface ProfessionalTaskOrigin {
  packageId: string;
  taskId: string;
  taskType: string;
}
```

该来源必须同步扩展到：

- `Canvas`。
- `SavedCanvas`。
- `RunRecord`。
- 运行副本创建和历史恢复逻辑。
- 保存、另存为、重新打开和迁移逻辑。

工作台根据任务记录显示：

```text
来自《雾港来信》 · 续写下一章    [返回小说家] [查看任务上下文]
```

普通画布没有来源，不显示任务条。

### 7.3 运行事件

master 需要统一发布：

```text
run-started
run-succeeded
run-failed
run-aborted
run-output-invalidated
```

不论运行由标题栏、姬子还是其他入口触发，专业任务都应收到同样事件。应用重启后还要根据运行历史恢复中断任务。

`agentRunner` 只发布通用结果和事件，不能导入小说家模块。

## 八、续写模板和结果

### 8.1 任务上下文快照

用户确认续写前，弹窗展示真正会送入画布的内容：

- 当前章正文。
- 当前章结尾摘要。
- 用户选择的人物和世界设定。
- 用户选择的未回收线索。
- 下一章目标、预计字数和写作要求。

确认后生成一份任务专属、不可变的 Markdown 上下文快照。快照属于专业任务，不随源章节后续编辑而改变；任务同时记录源章节修订号，若返回时源章节已变化要提示用户。

### 8.2 第一版使用两节点模板

```text
[续写作者：读取任务上下文快照]
              |
              v
[一致性检查与定稿：fictionist.chapter-draft]
```

不增加小说专属输入节点。根 Agent 通过 master 现有文件数据源读取任务上下文快照，第二个 Agent 接收上游完整正文并输出最终草稿。

模板不得固化某台机器的模型配置 ID。创建画布时由用户选择或解析当前可用模型；缺少模型时在创建前提示，不生成无法运行的画布。

### 8.3 稳定输出角色

最终节点声明：

```text
resultRole = "fictionist.chapter-draft"
```

运行结果至少提供：

```ts
interface RunOutputRef {
  nodeId: string;
  resultRole?: string;
  outputFormat: string;
  label: string;
  summary?: string;
  path?: string;
  dataPath?: string;
}

interface RunCanvasResult {
  runId: string;
  runCanvasId: string;
  nodeCount: number;
  writtenCount: number;
  outputs: RunOutputRef[];
}
```

`resultRole` 还必须写入运行历史和 `data.json`。小说家只接收唯一、非空、可读取的 Markdown `fictionist.chapter-draft`；缺失、重复、格式不符或产物被删除时进入明确失败状态。

## 九、首个真实用户流程

### 9.1 新建和保存

1. 用户在现有书架界面新建作品。
2. 用户进入正文页创建章节并编辑。
3. 保存时先写正文，再更新元数据。
4. 保存失败时保留未保存状态并显示错误。
5. 重启应用后恢复作品、章节、当前选择和待确认任务。

### 9.2 创建续写画布

1. 用户点击“续写下一章”。
2. 系统先保存当前章节；失败则停止。
3. 用户检查并选择上下文。
4. 小说家创建专业任务和上下文快照。
5. master 克隆两节点模板并生成新节点 ID。
6. 画布写入专业任务来源。
7. 应用切换到 master 工作台并打开新画布。
8. 用户可以检查、修改和保存该画布。

### 9.3 运行和返回

1. 用户使用 master 现有运行按钮。
2. master 正常显示节点状态、运行副本、历史和产物。
3. 运行事件把唯一章节草稿登记为待确认结果。
4. 工作台显示“草稿已生成，可返回小说家”。
5. 用户返回小说家预览或修改草稿。

### 9.4 确认保存

用户可以：

- 修改草稿。
- 放弃草稿。
- 重新打开对应画布。
- 保存为下一章。

保存为下一章时必须检查章序冲突、源章节修订和产物可用性。只有正文和索引都成功后，任务才变成 `accepted`。

现有小说家 UI 是正式视觉基线：

- 保留书架、正文、设定库、时间线和工作流整体布局。
- 保留当前颜色、密度、导航结构和主要文案。
- 只增加加载、空状态、保存中、保存失败、未保存、运行中和待确认状态。
- 功能接入不得顺带进行视觉重构。

`FictionistWorkspace.tsx` 已超过单文件维护范围。先补表征测试，再进行纯机械拆分；拆分和功能接入不得放进同一提交。

## 十一、分阶段实施

### 阶段 0：固定并拆分 UI 基线

- 为现有五个界面补关键渲染测试。
- 补书架进入正文、编辑和续写弹窗的交互表征测试。
- 按视图机械拆分大组件，不改布局和文案。

验收：测试、Lint、生产构建通过；三个桌面视口的人工检查与当前基线一致。

### 阶段 1：真实作品与章节

- 实现小说索引、章节仓储、schema 版本和恢复记录。
- 新建作品、新建章节、编辑、保存和重启恢复真实可用。
- 增加统一导航保护。
- 将小说动态键加入 hydration、数据扫描和清理逻辑。

验收：重启后数据存在；保存失败不能显示成功；离开页面不能静默丢稿。

### 阶段 2：通用专业任务与可见画布

- 增加通用任务记录、画布来源和运行事件。
- 扩展画布保存、运行副本和历史来源字段。
- 实现任务上下文快照和两节点模板。
- 从小说家创建并打开可编辑 master 画布。
- 工作台显示来源和返回入口。

验收：普通画布不变；专业来源经过保存、运行和重启后不丢失。

### 阶段 3：结果回传与确认保存

- 增加持久化 `resultRole` 和通用输出引用。
- 把成功、失败、中止和产物删除同步给任务。
- 在小说家显示待确认草稿。
- 实现修改、放弃、重新打开画布和原子保存下一章。

验收：只有用户确认后产生新章节；所有异常都不能生成空章节。

### 阶段 4：提取为可导入专业包

- 建立 `.mapp` 打包、导入、重启加载和失败回退。
- 小说家入口改由包注册结果生成，移除 master 中的硬编码。
- 小说家前端打成单入口 bundle。
- 验证禁用、升级和卸载的数据保留规则。

验收：干净 master 导入小说家包并重启后，首个闭环与分支版本一致。

### 阶段 5：扩展小说能力

- 设定条目和时间线真实增删改查。
- 建立章节、人物、地点、事件引用。
- 增加润色、审校、设定提取和线索检查。
- 自动提取结果先进入候选区，用户确认后写入正式设定。

- 小说索引版本和迁移。

### 14.2 集成测试

- 新建作品和章节后重启恢复。

- 小说家创建任务后打开正确画布。

- 接收中断后可以恢复，不产生孤立章节。
- 干净 master 导入 `.mapp` 后获得相同功能。
- 同时安装多个专业包时入口、工具和数据不冲突。

## 十五、主要风险

### 风险 1：分支功能无法提取成包

处理：开发阶段就通过公开入口和 master 门面接入；以“干净 master 导入包”作为最终门禁。

### 风险 2：小说逻辑污染运行器

处理：运行器只发布通用任务事件和输出引用，禁止导入小说家模块。

### 风险 3：专业来源在快照中丢失

处理：同时扩展画布、已保存画布、运行副本和运行历史，并补保存/恢复测试。

### 风险 4：结果识别错误

处理：使用持久化唯一 `resultRole`，不根据名称、顺序或文件名猜测。

### 风险 5：章节被自动覆盖

处理：结果先进入候选草稿；用户确认后才执行带恢复记录的写入。

### 风险 6：动态章节数据无法清理

处理：新增服务端受控命名空间扫描和清理，不能只维护前端键列表。

### 风险 7：可执行包影响应用

处理：安装时明确完全信任风险；重启加载；加载失败自动禁用或回退；提供不加载专业包的安全启动方式。

### 风险 8：功能接入破坏认可 UI

处理：先建立 UI 表征测试并单独机械拆分，功能提交不包含无关视觉重构。

## 十六、完成标准

首个小说家闭环必须同时满足：

- 用户创建的作品和章节重启后仍存在。
- 离开小说家不会静默丢失正文。
- “续写下一章”创建并打开真实、可编辑的 master 画布。
- 画布明确显示来自哪部作品和哪个章节。
- 用户可以修改节点并使用 master 现有运行流程。
- 系统只接收唯一的 `fictionist.chapter-draft` Markdown 输出。
- 草稿可以返回小说家预览和修改。
- 用户确认后才保存为下一章。
- 运行失败、中止或产物缺失不会生成空章节。
- 普通工作台和普通画布行为不变。
- 小说家现有 UI 风格和布局不退化。
- 小说家可以被打包为 `.mapp`。
- 干净 master 导入并重启后获得相同功能。
- 小说家被禁用或卸载后，master 和其他专业包正常运行。

## 十七、暂不实施

- 多人协作和云同步。
- Git 式小说版本分支和合并。
- 自动改写正式设定库。
- 全书向量数据库和语义检索。
- 专业包在线市场和自动更新。
- 专业包热加载。
- 包内 Rust 扩展。
- 不可信包安全沙箱。
- 手机和平板布局。
- 一次性实现所有小说工作流。

这些能力应在首个续写闭环和 `.mapp` 导入验证稳定后分别立项。
