# 开发错误总结（2026-07-07 ~ 07-08）

> 沉淀 v2.94 多模态图像输入链路 + v2.95 门控节点 + v2.96 Token 统计 + v2.97 体验修复过程中犯的错误与教训，避免下次重蹈。

## 一、多模态链路 v2.94

### 1.1 read_rich 的 comments 与 rows 不对齐（已修）

**现象**：`read_rich` 在 `sheet_name=None` 时，`rows` 取 `wb.active`，但 `_extract_comments(wb, None)` 扫**全部** worksheet 的批注。text 里会混入其他表的批注却无对应 rows——模型会困惑。

**根因**：`_extract_comments(wb, sheet_name)` 的 `sheet_name=None` 语义是「全部表」，但 `read_rich` 的 rows 只取一张表，两者口径不一致。

**修法**：`read_rich` 改传 `ws.title`（与 rows 同表），而非原始 `sheet_name`。

**教训**：富读取函数把多个数据维度（rows/comments/images）聚合时，**每个维度的「作用范围」必须一致**。rows 取一张表，comments 也只能扫同表。不能因为函数签名支持「全部」就无脑传 None。

### 1.2 _extract_comments 缺 limit_rows 护栏（已修）

**现象**：`_extract_comments` 用 `ws.iter_rows()` 不带 `max_row`，对超大 Excel（10万行）全表遍历。而 `_read_rows` 有 `limit_rows=10000` 护栏。两者不一致。

**根因**：写 comments 提取时只想着「批注稀疏，全扫无所谓」，忘了和 rows 的行数护栏对齐。

**修法**：`_extract_comments` 加 `limit_rows` 参数，用 `ws.iter_rows(max_row=...)`。

**教训**：同一 action 内的多个数据维度，**行数护栏要统一**。不能一个有限制一个没限制——超大文件时一个维度会把整个请求拖慢。

### 1.3 inputs.ts 无意义三元（已修）

**现象**：`executeTool(path ? 'excel' : 'excel', {...})` —— 三元的两个分支都是 `'excel'`，无意义。

**根因**：写代码时手滑，本想写条件分派但写错了。

**修法**：直接 `executeTool('excel', {...})`。

**教训**：写完三元立刻检查两个分支是否真不同。**无意义三元是代码噪音**，review 时一眼就该发现。

## 二、门控节点 v2.95

### 2.1 ready 判定没考虑上游 failed/skipped（核心设计漏洞，Plan agent 发现）

**现象**：最初方案里 `onNodeSuccess` 只在上游 success 时推进下游 ready。但门控节点需要上游 failed/skipped 也推进——否则 AND 上游有 failed 时节点永远不 ready（卡死）。

**根因**：设计时只想着「上游成功→下游可以跑」的正向路径，没想「上游失败→门控该判不通过」的反向路径。原 `parents.every(success)` 判定对 agent 节点是对的（agent 上游失败就该跳过下游），但对门控节点是错的（门控要等所有上游 settled 才能判，不能只等 success）。

**修法**：抽 `isReady(childId)` —— 门控 ready = 所有上游 settled（success/failed/skipped），agent ready = 所有上游 success（原逻辑）。`onNodeSuccess` 改名 `onNodeSettled`，在 success/failed/skipped 三种 settled 状态都推进下游。

**教训**：**「上游成功推进下游」和「上游失败推进下游」是两个不同的调度语义**。agent 节点要前者（失败就跳过下游），门控节点要后者（失败也要算 settled 才能判布尔值）。新增节点类型时，要问自己：这个节点的 ready 判定要不要认 failed/skipped？

### 2.2 markSkipped 与 onNodeSettled 的调用顺序（自检发现，已修）

**现象**：catch 分支 L468 先 `markSkipped(所有下游)`（递归标 skipped），L472 才 `onNodeSettled(node)`。但 markSkipped 内部 L271 已经调了 `onNodeSettled(node)`——导致 onNodeSettled 被调两次。且 markSkipped 递归标下游时，onNodeSettled 在递归中途就被调用，此时下游还没全部标 skipped，可能 push 即将被标 skipped 的节点进 ready 队列。

**根因**：markSkipped 是递归函数，内部调 onNodeSettled 是为了「skipped 算 settled，可能让 NOR 下游 ready」。但 catch 分支外部又调一次，冗余且顺序不对。

**修法**：catch 分支保留 `onNodeSettled(node)`（在 markSkipped 递归后），靠 `readyHas` Set 防重复 push。顺序：先 markSkipped 递归标完所有该跳过的下游，再 onNodeSettled 扫直接下游。

**教训**：**递归函数内部触发回调时，调用方要注意是否会重复触发**。防重靠 Set/标志位，但顺序也要对——先把该标的状态标完，再触发「状态变化后该推进谁」的扫描。

### 2.3 辅助函数定义位置（已修）

**现象**：`isGate`/`isReady`/`readyHas`/`onNodeSettled` 最初定义在 `await new Promise` 内部（L283 之后），但初始扫描在 L206 之前，用不到这些函数。

**根因**：写代码时先写了初始扫描，后加门控逻辑时把辅助函数塞进了 Promise 内部，没意识到初始扫描也要用。

**修法**：把这四个辅助定义提到 `let aborted` 之后（L200 之后），在初始扫描之前。

**教训**：**辅助函数要定义在使用点之前**。JS 函数声明会提升，但 `const` 箭头函数不提升——定义在后面用在前面会报错。即使提升，逻辑上定义在使用前也更清晰。

## 三、命名与 UI 设计教训

### 3.1 命名用了英文 OR/AND/NOR（用户要求改中文）

**现象**：最初 GateNode 默认 label 用「OR 门」「AND 门」「NOR 门」，命令面板/属性面板都英文。用户反馈要改「与或非」。

**根因**：设计时直接用了逻辑学符号 OR/AND/NOR，没考虑中文用户习惯。

**修法**：改「或门」「与门」「非门」，内部 `gateType` 值仍 `'or'/'and'/'nor'`（不变）。

**教训**：**中文项目里用户可见名一律用中文**。内部标识用英文（避免拼音），但展示给用户的名字要中文。逻辑学术语「与/或/非」是中文里的标准说法，比「AND/OR/NOR」更亲切。

### 3.2 UI 丑陋（用户反馈后重做）

**现象**：最初 GateNode 用「菱形 + border-radius:4px 没旋转 + 文字 ∨/∧/⊽ 图标」，用户直言「这个UI好丑」。

**根因**：
1. 菱形其实没真做（只写 border-radius:4px，没 transform:rotate）
2. 文字 ∨/∧/⊽ 太简陋，不同字体下渲染不一致
3. 配色用 #fafafa 底 + #8c8c8c 边，太灰不抢戏也不精致

**修法**：保持圆角矩形（和 agent 节点统一风格，连接点位置不怪）+ 左侧 4px 彩色竖条（类型标识）+ SVG 精致门控符号（∪/∩/∪带否定横线）+ 浅 tint 底色 + 对应色边框。

**教训**：
1. **菱形节点在 React Flow 里连接点位置会怪**（Handle 在节点边缘，菱形边缘不在矩形边）——别为了「不一样」搞菱形，圆角矩形更稳。
2. **文字符号（∨∧⊽）在不同字体下渲染不一致**——用 SVG 画图标更精致可控。
3. **节点视觉区分用「左侧竖条 + 浅 tint 底色」比「整块染色」更克制**——和 agent 节点同框时不抢戏。

### 3.3 非门语义没在设计阶段对齐（用户要求屏蔽）

**现象**：用户最后要求屏蔽非门，理由「语义需求我还没有彻底明确」。

**根因**：设计时我直接把 NOR 当作「非门」的实现（NOR=全部非 success→通过，单输入退化为 NOT）。但**多输入 NOR 与单输入 NOT 的语义差异**没在设计阶段和用户对齐——多输入时「非」到底指「或非」还是「全否定」？用户没想清楚就让我做，我也没主动问。

**修法**：从命令面板移除非门入口，但保留编排器/GateNode/属性面板/导入导出的非门支持代码——等用户明确语义后把命令加回来即可。

**教训**：**设计阶段对「语义模糊点」要主动追问**。用户说「非门」时，我应该立刻问：「这是单输入 NOT（单个不通过→通过），还是多输入 NOR（全部不通过→通过）？多输入时叫『非』是否严谨？」而不是直接实现 NOR 当「非门」。

## 四、工具使用错误

### 4.1 Write 工具连续空参数调用（操作错误）

**现象**：写 plan 文件时连续 4 次 `Write` 工具调用都没传 `file_path`/`content` 参数，全部报 InputValidationError。

**根因**：我以为 Write 能直接覆盖旧 plan 文件，但调用时参数没带上。

**修法**：改用 `Edit` 工具的 `old_string`/`new_string` 全量替换，绕过 Write 的参数问题。

**教训**：**Write 工具必须带 `file_path` 和 `content` 两个必填参数**。如果只是修改已有文件，优先用 Edit（只发 diff）；只有创建新文件或完全重写时才用 Write。调用前检查参数是否齐全。

## 四·五、异步 UX 与 dev 体验（v2.97）

### 4.5.1 异步操作前没先回显用户输入 + 加载态（已修）

**现象**：给姬子发「创建画布/节点」类指令，UI 先卡十几秒才弹确认框——用户消息不先显示、没有「思考中」提示，一点不丝滑。

**根因**：`MasterAgentPanel.tsx` `onSend` 的**规划分支把顺序搞反了**——先 `await planMasterActionWithLLM(...)`（十几秒的 LLM 调用），返回后才 `addMessage` 用户消息。而同文件的聊天分支和造工具分支都是正确顺序：**先回显用户消息 + 加 `status:'sending'` 思考占位，再 await**。同一个文件里三条分支两条对一条错，是复制粘贴时漏改了顺序。

**修复**：规划分支 await 前先 `addMessage` 用户消息 + `setDraft('')` + 加思考占位（记 `planAssistantId`），await 返回后用 `updateMessage` 把占位改成确认说明（复用而非新增）。聊天回落路径用 `planUserMsgAdded`/`planAssistantId` 两个跨块变量避免重复回显。

**教训**：**任何耗时异步操作（LLM 调用、网络请求、文件 IO）启动前，必须先同步更新 UI——回显用户输入 + 显示加载态**。顺序永远是「先反馈、再 await」。异步不阻塞主线程不代表 UI 会自己更新——await 前没 setState，界面就一直是旧状态干等。同一组分支里若有一条是正确范式，其余分支照抄它的顺序。附带好处：加载占位让「发送中」状态可见，能顺便拦住重复提交。

### 4.5.2 Vite dev 未配 optimizeDeps.include 导致启动白屏（已修）

**现象**：每次 `tauri dev` 启动都白屏一段时间才出界面。

**根因**：`vite.config.ts` 没有 `optimizeDeps.include`。dev 模式下 Vite 按需发现依赖并预打包，若首次加载途中「发现」新依赖会触发**重新预打包 + 整页重载**（日志 `Re-optimizing dependencies because vite config has changed`），表现为白屏。

**修复**：把全部一级重量级依赖（react/react-dom/zustand/antd/@ant-design/icons/@xyflow/react/@tauri-apps/*）显式列进 `optimizeDeps.include`，预打包确定化。

**教训**：
1. **dev 体验问题要和交付质量问题分开看**——这个白屏**只影响开发调试**（Vite 按需编译的固有代价），**正式打包（NSIS）是预打包静态文件、秒开、完全不受影响**。诊断时先分清「是 dev-only 还是会带到成品」，别把 dev 固有现象当成产品 bug 过度投入。
2. React + antd + @xyflow/react 这类大依赖图的项目，`optimizeDeps.include` 显式列一级依赖是标准优化，避免加载途中二次优化重载。
3. 改 `vite.config.ts` 必须重启 dev server 才生效（热更新不覆盖 config）。

## 五、通用教训

### 5.1 「富读取」类函数要保证各维度口径一致

`read_rich` 把 rows + comments + images 聚合时：
- rows 取 active 表 → comments 也要扫 active 表（不能扫全部）
- rows 有 limit_rows → comments 也要有 limit_rows

**规则**：多维度聚合函数，每个维度的「作用范围」和「护栏」要一致。

### 5.2 新增节点类型要问 ready 判定认哪些状态

agent 节点 ready = 所有上游 success（失败就跳过下游）。
门控节点 ready = 所有上游 settled（失败也要算，才能判布尔值）。

**规则**：新增节点类型时，问自己：这个节点的 ready 判定要不要认 failed/skipped？如果认，是不是要抽 `isReady(childId)` 按类型分流？

### 5.3 递归函数内部触发回调时调用方注意顺序与防重

markSkipped 递归标 skipped，内部调 onNodeSettled 推进下游。调用方（catch 分支）又要调 onNodeSettled——顺序要先递归完再调，防重靠 Set。

**规则**：递归函数内部触发「状态变化回调」时，调用方在递归结束后也要再调一次（防漏），靠标志位防重复触发。

### 5.4 中文项目用户可见名用中文

内部标识英文（`gateType: 'or'`），展示名中文（「或门」）。

**规则**：UI 文案、命令面板、属性面板等用户可见处一律中文；代码标识、API 字段、数据库字段用英文。

### 5.5 设计阶段主动追问语义模糊点

用户说「非门」时，要问清单输入还是多输入、多输入时语义是什么。不要直接实现一个自以为是的版本。

**规则**：遇到语义可能有多解的词（「非」「或」「跳过」「通过」），立刻用 AskUserQuestion 追问，不要猜。

### 5.6 UI 视觉区分用「左侧竖条 + 浅 tint」而非「整块染色」

节点视觉区分时，左侧 4px 彩色竖条 + 对应色浅 tint 底色，比整块染色更克制，和主节点同框时不抢戏。

**规则**：节点类型区分用「左侧竖条 + 浅 tint 底色 + 对应色边框」，不用「整块染色 + 粗边框」。

## 六、自检 checklist（下次开发新功能时过一遍）

- [ ] 富读取/聚合函数：各维度的「作用范围」是否一致？（同表、同行数上限）
- [ ] 新增节点类型：ready 判定要不要认 failed/skipped？
- [ ] 递归函数 + 状态回调：调用方顺序是否对？防重标志位是否齐？
- [ ] 用户可见名：是否中文？内部标识是否英文？
- [ ] 语义模糊点：是否在设计阶段追问清楚？
- [ ] UI 视觉区分：是否用「左侧竖条 + 浅 tint」而非「整块染色」？
- [ ] Write 工具：参数是否齐全？能否用 Edit 替代？
- [ ] 三元表达式：两个分支是否真不同？
- [ ] 辅助函数：定义是否在使用点之前？
- [ ] 行数护栏：同 action 内各维度是否统一？
- [ ] 异步操作：await 前是否先回显用户输入 + 加载态？同组分支顺序是否一致（先反馈再 await）？
- [ ] 体验问题：是 dev-only（Vite 按需编译等固有代价）还是会带到正式包？别把 dev 现象当产品 bug。
- [ ] 改了 Python 工具的返回结构/行为：是否同步升 `LLM_CALLING_VERSION` + 前端 `EXPECTED_PYTHON_SERVICE_VERSION`？（不升→旧后台不重启→跑旧代码，功能看似坏了）
- [ ] 安全边界：是否靠服务端硬拦而非前端约定？（任何人可直接打 localhost API，前端提示≠防护）
