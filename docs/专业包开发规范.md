# 专业包开发规范

> 状态：现行规范
>
> 制定日期：2026-07-23
>
> 适用范围：`master` 基础项目、`fictionist` 分支及后续所有专业包
>
> 目标：让后来接手的开发者先理解产品意图，再决定代码结构和实现方式。

## 一、先理解产品是什么

本项目最终不是一个只服务某个职业的应用，而是一个可以安装多个专业包的多 Agent 工作平台。

```text
master
  = 基础工作台
  = 多 Agent 画布、节点、模型、工具、运行历史和专业包管理能力

fictionist
  = 基于 master 开发的第一个专业包
  = 小说作品管理界面 + 小说任务模板 + 输入整理 + 结果接收

未来的测试工程师等专业包
  = 使用同一个 master 画布完成其他职业任务
```

专业包不是另一套独立的 Agent 软件。它负责理解某个专业领域，真正的 AI 协作、节点编排和任务运行仍由 master 完成。

## 二、已经确定的产品约定

以下约定已经确认，后续开发不得自行改成另一种产品方向。

1. `master` 是基础项目，不包含小说家、测试工程师等具体职业的业务实现。
2. 专业包基于 master 提供的画布、Agent、模型和工具能力工作。
3. 一个 master 可以同时安装多个专业包，并在不同专业工作区之间切换。
4. 专业包可以拥有自己的界面、数据、模板、前端逻辑和 Python 工具。
5. 第一版允许用户导入本地可执行专业包；用户确认后，包代码获得与应用近似的执行能力。
6. 第一版不为专业包提供安全沙箱，也不对恶意包作隔离保证。
7. 导入、升级、禁用或卸载专业包后，重启应用才生效。
8. 第一版专业包可以包含前端 JavaScript 和 Python 工具，不支持包内 Rust 原生扩展。
9. 专业包创建 AI 任务后，必须先打开一张可见、可编辑的 master 画布，由用户检查后运行。
10. 专业包不能建立自己的 Agent 运行器，也不能绕过 master 直接复制一套画布系统。
11. 普通画布没有专业包来源，行为必须保持不变。
12. 专业包输出先进入待确认状态，不能自动覆盖正式专业数据。

## 三、职责边界

### 3.1 master 负责什么

master 负责所有专业包共同依赖的基础能力：

- 多 Agent 画布及节点编辑。
- 画布保存、运行副本、运行历史和产物管理。
- 模型配置、工具调用、Python 后台和文件能力。
- 专业包导入、安装、启用、禁用、升级、回退和卸载。
- 专业包入口展示和工作区承载。
- 专业任务与画布运行之间的通用关联。
- 运行开始、成功、失败和中止等通用事件。
- 带命名空间的专业包存储和数据清理能力。
- 应用启动失败时的专业包安全恢复入口。

master 不负责理解“章节”“测试用例”“病历”等专业概念。

### 3.2 专业包负责什么

每个专业包负责自己的专业领域：

- 专业数据及其编辑界面。
- 把用户意图转换成专业任务。
- 从专业数据中选择和整理任务上下文。
- 提供适合该任务的画布模板。
- 识别 master 返回的专业结果。
- 让用户预览、修改和确认结果。
- 确认后将结果写入专业数据。
- 自己的数据升级、导出和清理说明。

### 3.3 禁止的耦合

专业包不得：

- 直接导入 `canvasStore`、`agentRunner` 或其他 master 内部实现。
- 修改普通画布的业务规则来迁就某个专业包。
- 在 master 中加入只对某一个专业包有意义的字段或分支。
- 把专业正文、测试报告等大块专业数据塞进画布主存储。
- 直接调用模型完成专业任务而不留下可追溯的画布运行记录。
- 删除不属于自己的画布、Agent、工具或其他包的数据。

如确实需要新的基础能力，应先把它设计成所有专业包都能使用的 master 能力，再由专业包调用。

## 四、专业包与画布的核心关系

所有需要多 Agent 协作的专业任务都遵循同一条链路：

```text
用户在专业工作区发起任务
  -> 专业包整理专业数据和要求
  -> 专业包创建一条可追溯的专业任务
  -> master 根据包模板创建可见画布
  -> 用户查看或修改节点
  -> 用户点击 master 的运行按钮
  -> master 运行画布并保存历史与产物
  -> master 将结果通知对应专业任务
  -> 专业包展示候选结果
  -> 用户确认后写入正式专业数据
```

专业工作区用于管理专业内容，master 工作台用于编辑和运行流程。两套界面保持独立，但通过专业任务形成强关联。

第一版不提供隐藏画布的一键运行。未来可以增加快捷运行，但底层仍必须创建画布和运行记录，并允许用户随时打开查看。

## 五、专业任务的通用规则

### 5.1 画布只保存最小来源信息

画布、已保存画布、运行副本和运行历史都应保存同一份通用来源：

```ts
interface ProfessionalTaskOrigin {
  packageId: string;
  taskId: string;
  taskType: string;
}
```

作品 ID、章节 ID、测试项目 ID、返回页面等详细信息由专业任务保存，不进入通用画布类型。

来源信息必须在以下过程中完整保留：

- 保存和重新打开画布。
- 创建运行副本。
- 写入运行历史。
- 应用重启和异常恢复。
- 另存为用户副本。

若用户明确复制为普通画布，可以去除专业任务关联；系统必须提示复制后的结果不会自动返回原专业任务。

### 5.2 运行生命周期

master 必须向专业任务提供通用运行事件：

```text
已创建 -> 待运行 -> 运行中 -> 待确认 -> 已接收
                         |          |
                         v          v
                       失败       已放弃
```

至少要处理：

- 用户开始运行。
- 运行成功。
- 运行失败。
- 用户中止。
- 应用异常退出。
- 产物被删除或无法读取。
- 失败节点被重新运行。

不能只把成功结果返回给运行按钮。专业任务需要在所有调用入口和应用重启后恢复到可理解状态。

### 5.3 输出身份

专业包不能根据节点名称、文件名、产物顺序或“第一个输出”猜测结果。

最终节点使用稳定的输出角色，例如：

```text
fictionist.chapter-draft
tester.test-report
```

输出角色必须同时保存在：

- 节点定义。
- 运行返回结果。
- 运行历史。
- 节点 `data.json` 产物信封。
- 专业任务的结果引用。

同一任务要求唯一结果时，找不到、找到多个、格式不符或产物缺失都必须明确失败，不能静默选择。

## 六、专业包的用户体验

### 6.1 安装多个专业包

用户可以同时安装小说家、测试工程师等多个专业包。安装后，master 在统一位置展示各专业工作区入口。

专业包入口由安装结果生成，不允许在 `App.tsx`、`TitleBar.tsx` 等主入口中长期写死具体包名。

### 6.2 导入

第一版导入流程：

```text
选择专业包文件
  -> 展示名称、版本、能力和数据影响
  -> 明确提示“该包将运行本地代码”
  -> 用户确认完全信任
  -> 校验包结构和兼容版本
  -> 安装到独立目录
  -> 提示重启应用
```

导入失败不能留下半安装状态。已经存在同版本时应给出明确选择，不能直接覆盖。

### 6.3 启用和禁用

- 启用、禁用都在重启后生效。
- 禁用只移除入口和能力，不删除专业数据。
- 有正在运行的专业任务时，不允许禁用或卸载对应包。
- 包加载失败时，master 应禁用该包并保留其他包和基础工作台。

### 6.4 卸载

卸载分为包代码和用户数据两部分：

1. 移除包代码、模板和包提供的 Python 工具。
2. 让用户选择专业数据如何处理：
   - 默认保留数据，重新安装后恢复。
   - 导出后删除。
   - 直接删除，必须二次确认。

卸载任何专业包都不能删除普通画布、普通 Agent、其他包数据和基础设置。

## 七、包格式和运行边界

### 7.1 文件格式

专业包使用单文件分发，例如：

```text
fictionist-1.0.0.mapp
```

`.mapp` 是压缩包，至少包含：

```text
manifest.json          包名称、ID、版本和兼容要求
frontend/index.js      单入口前端 bundle
python/tools/          可选的 Python 工具
assets/                图标和静态资源
migrations/            可选的数据迁移
schemas/               可选的数据与结果格式说明
```

包 ID、内部文件路径和存储键使用稳定的 ASCII 标识；用户可见名称使用中文。

### 7.2 信任模型

第一版采用“用户完全信任本地包”的模型：

- 允许导入用户选择的本地可执行包。
- 包前端代码运行在应用权限范围内。
- 包 Python 工具运行在现有 Python 后台内。
- 包可能读取或修改本地数据、访问网络并影响应用稳定性。
- 安装确认必须把这些风险写清楚。
- 包声明的能力用于告知、审计和兼容检查，不是安全隔离。

不得在产品文案中把第一版描述为“安全沙箱”“权限隔离”或“恶意包无法影响系统”。

### 7.3 第一版支持范围

支持：

- 前端 JavaScript 专业界面和业务逻辑。
- 专业画布模板和 Agent 配置。
- 专业包自带 Python 工具。
- 静态资源、数据 schema 和迁移脚本。

不支持：

- 包内 Rust 原生扩展或动态 DLL。
- 安装后立即热加载。
- 在线市场和自动下载更新。
- 对不可信第三方代码的安全隔离。

需要新的 Rust 能力时，应先在 master 中实现稳定的通用能力，再通过宿主接口提供给专业包。

## 八、存储、数据和迁移

### 8.1 独立命名空间

每个专业包必须拥有唯一存储命名空间，例如：

```text
multi-agent-fictionist-*
multi-agent-tester-*
```

专业包只能通过 master 提供的带命名空间存储门面读写自己的数据。命名空间是防止普通开发错误误写其他数据的边界，不是对恶意包的安全隔离。

### 8.2 大内容分离

大块专业正文不能全部放入一个 Zustand JSON。应拆成元数据索引和独立内容记录。例如小说章节按章保存，测试日志按任务或运行保存。

存储字段需要明确：

- `schemaVersion`。
- 稳定 ASCII ID。
- 创建和更新时间。
- 修订号或内容哈希。
- 数据归属包 ID。

可由 ID 推导的存储键不要重复写进业务实体，避免 ID 与键不一致。

### 8.3 原子保存和恢复

跨多个文件的专业写入需要恢复记录。典型流程：

```text
记录待执行变更
  -> 写入正文或结果
  -> 更新元数据索引
  -> 标记任务完成
  -> 清除恢复记录
```

应用启动时检查未完成记录并恢复，不能产生正文已写入但索引找不到的孤立数据。

### 8.4 数据清理

桌面端基础数据仍按固定分类清理，专业包动态数据必须增加受控命名空间扫描。小说家已经实现 `multi-agent-fictionist-*` 的 `.json`、`.bak` 和临时残留清理；后续新增专业包时仍必须同步扩展：

- 数据占用扫描。
- 精确命名空间识别。
- `.json`、`.bak` 和残留临时文件处理。
- 设置中心的影响说明。
- 删除范围回归测试。

仅在前端声明 `ownedStoragePrefixes` 不代表已经实现清理。

## 九、升级和失败恢复

### 9.1 版本兼容

专业包清单至少包含：

- 包 ID。
- 包版本。
- 所需 master 专业包接口版本。
- 前端入口。
- Python 工具清单。
- 存储命名空间。
- 数据迁移版本。
- 用户可见的能力和风险说明。

接口版本不兼容时拒绝启用，并告诉用户应该升级 master 还是更换包版本。

### 9.2 升级和回退

- 新版本先安装到独立版本目录，不覆盖当前可用版本。
- 重启后尝试加载新版本。
- 加载或迁移失败时回退到上一版本。
- 数据迁移执行前必须备份或提供反向恢复方案。
- 包升级不能覆盖用户复制和修改过的画布模板。

### 9.3 安全启动

如果某个包导致应用连续启动失败，master 应提供不加载专业包的安全启动方式，并优先禁用最近安装或升级的包。

安全启动是稳定性恢复机制，不是恶意代码隔离机制。

## 十、界面与交互规范

1. master 保持统一的一级工作区切换方式，专业包入口由已启用包生成。
2. 每个专业包可以有自己的领域界面，但不能覆盖 master 全局样式。
3. 专业包发起任务后必须打开 master 工作台中的真实画布。
4. 工作台应显示任务来自哪个专业包和专业对象。
5. 普通画布不显示专业任务条。
6. 专业数据有未保存修改时，离开前必须保存或明确确认放弃。
7. 切换回专业工作区不能中止正在运行的画布。
8. 运行完成后，用户可以在工作台查看产物，也可以返回专业工作区确认。
9. 用户可见名称和提示使用中文；内部 ID、包 ID、任务类型和输出角色使用英文 ASCII。
10. 专业包 UI 已经形成认可基线时，功能接入不得顺带做无关视觉重构。

## 十一、分支和开发流程

### 11.1 分支定位

- `master`：基础项目和专业包宿主能力的事实源。
- `fictionist`：基于 master 开发的小说家专业包及首个专业包验证分支。
- 后续专业包可以有各自分支，但都必须以兼容的 master 为基础。

### 11.2 开发阶段也要遵守包边界

专业包仍在分支源码内开发时，可以采用内置开发加载方式，但必须走与导入包相同的注册、存储、任务和画布门面。

禁止为了开发方便长期保留：

- master 直接导入某个专业包组件。
- 标题栏直接写死专业包名称。
- 专业包直接调用 master 内部 store。
- 只有分支内可以运行、打包导入后无法运行的特殊路径。

### 11.3 提交边界

master 通用能力和专业包业务应尽量分开提交。例如：

```text
功能：增加专业任务运行事件
功能：增加专业包命名空间存储
功能：小说家接入续写任务事件
```

不要在同一个提交中同时进行宿主重构、专业业务开发、UI 大改和数据迁移。

### 11.4 专业包提取验收

小说家开发完成后，必须验证：

1. 干净 master 不包含小说家入口和小说业务代码。
2. 干净 master 可以导入小说家 `.mapp`。
3. 重启后出现小说家入口。
4. 导入后的功能与 `fictionist` 分支验收结果一致。
5. 禁用或卸载小说家后，master 与其他专业包正常运行。

## 十二、建议目录边界

目标目录可按下面的职责逐步迁移，不要求一次性重排现有仓库：

```text
app/src/
  professionalHost/       master 的包管理、注册和通用任务门面
  stores/canvas/          master 画布状态
  lib/agentRunner/        master 运行器

packages/
  fictionist/
    manifest.json
    src/
      index.ts             对 master 暴露的唯一入口
      domain/              作品、卷、章节等专业类型与规则
      application/         续写、接收草稿等用例
      infrastructure/      小说数据仓储
      ui/                  现有小说家界面
      templates/           专业画布模板
      python-tools/        可选 Python 工具
    tests/
```

专业包外部只能导入它的公开入口；内部文件不得被 master 深层引用。

## 十三、测试和验收门禁

### 13.1 master 基础回归

- 未安装任何专业包时可以正常启动和使用普通画布。
- 安装多个专业包后可以正常切换。
- 某个包加载失败不影响其他包和基础工作台。
- 普通画布运行、历史和产物行为不变。

### 13.2 包生命周期

- 导入损坏包时没有半安装残留。
- 不兼容版本被明确拒绝。
- 安装、升级、禁用和卸载均在重启后生效。
- 新版本加载失败可以回退。
- 安全启动可以跳过所有专业包。
- 卸载默认保留用户专业数据。

### 13.3 专业任务

- 专业包可以创建并打开真实画布。
- 任务来源经过保存、运行和重启后不丢失。
- 成功、失败、中止和产物删除都能同步到专业任务。
- 找不到唯一专业输出时不能写入正式数据。
- 用户确认前不能自动覆盖专业数据。

### 13.4 验证命令

前端或 master TypeScript 变更至少执行：

```powershell
cd app
npm.cmd test
npm.cmd run lint
npm.cmd run build
```

涉及 Rust 存储或包安装命令时追加：

```powershell
cd app\src-tauri
cargo test
cargo clippy -- -D warnings
```

涉及 Python 工具加载和注册时追加：

```powershell
cd python
.\venv\Scripts\python.exe -m pytest -q
```

## 十四、首个专业包的实施范围

小说家是专业包体系的第一个验证项目。首个完整闭环跨越《小说家与工作台集成规划》的阶段 1–3，依次打通：

```text
新建作品
  -> 创建和编辑章节
  -> 保存并重启恢复
  -> 续写下一章
  -> 打开可编辑的 master 画布
  -> 用户运行画布
  -> 草稿返回小说家
  -> 用户确认后保存为下一章
```

小说家现有书架、正文、设定库、时间线和工作流界面作为视觉基线。首个完整闭环只增加必要的真实数据、加载、错误、运行中和待确认状态，不做无关视觉重构。

设定库和时间线在首个完整闭环期间继续保留演示状态，等核心链路通过后再真实化。

## 十五、交接检查清单

后来接手的开发者开始工作前，应逐项确认：

> 这是每次专业包开发和交接都要重新确认的门禁清单，不表示当前小说家阶段的完成进度，因此保持未预先勾选。

- [ ] 我理解专业包不是独立 Agent 应用，而是使用 master 画布的专业工作区。
- [ ] 我没有在 master 中写死具体专业包名称或组件。
- [ ] 我没有让专业包直接导入 master 内部 store 或运行器。
- [ ] 新任务会创建可见、可编辑且可追溯的画布。
- [ ] 专业来源会保留到保存画布、运行副本和运行历史。
- [ ] 专业输出有稳定角色，不靠名称或顺序猜测。
- [ ] 专业结果在用户确认前不会写入正式数据。
- [ ] 新存储已加入 hydration、迁移、数据清理和卸载测试。
- [ ] 包前端与 Python 工具均有唯一命名空间。
- [ ] 包变更在重启后生效，失败有回退或安全启动方式。
- [ ] 当前认可的专业 UI 没有被功能接入顺带重做。
- [ ] master 无包、单包和多包三种状态都完成回归验证。

## 十六、决策记录

2026-07-23 已确认：

- master 是基础项目，专业包基于 master 开发。
- 小说家完成后要能从分支提取成可导入包。
- 导入后的小说家功能应与当前 `fictionist` 分支一致。
- 后续允许存在测试工程师等多个专业包。
- 允许同时安装多个专业包。
- 专业包必须使用 master 多 Agent 画布完成专业任务。
- 第一版允许用户导入并完全信任本地可执行包。
- 包变更重启后生效。
- 第一版支持前端 JavaScript 和 Python 工具，不支持包内 Rust 扩展。
- 专业任务先打开可见、可编辑画布，再由用户运行。
- 小说家现有 UI 作为视觉基线。
- 小说家阶段 1 已完成作品、章节、正文保存、重启恢复、离开保护和动态数据清理。
- 小说家首个完整闭环后续由阶段 2 完成续写画布、阶段 3 完成草稿回传与确认保存。

任何改变上述方向的开发都应先更新本规范并明确记录原因，不能只在代码中形成隐含的新规则。
