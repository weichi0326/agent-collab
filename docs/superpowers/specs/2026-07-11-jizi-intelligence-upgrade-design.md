# 姬子智能升级设计

日期：2026-07-11

## 目标

将姬子从依赖单次规划和固定动作的产品内助理，升级为能够在明确安全边界内观察项目、制定有限计划、执行受控动作、验证结果并在失败时回滚或重新规划的智能助理。

本次升级覆盖现有动作，并扩展 Agent 配置修改、画布删除、自定义工具覆盖和删除。普通写操作需要一次确认，删除操作需要二次确认。任何写操作都不能由模型直接修改 Store 或文件。

## 设计原则

1. 证据优先：规划、诊断和验证只能依据真实项目状态与运行结果。
2. 人工控制：读取、分析和验证可自动进行；写操作必须确认；删除必须二次确认。
3. 事务执行：执行前预检并创建快照；失败时优先恢复执行前状态。
4. 有限自治：最多执行 8 步，单步最多修正 1 次，最多重新规划 2 次。
5. 渐进兼容：保留现有画布、Agent、工具、会话、Skill 和记忆数据，通过版本化迁移升级。
6. 测试先行：每项行为先写失败测试，再实现最小改动，最后运行全量验证。

## 非目标

- 不允许模型执行任意代码或直接调用 Zustand Store。
- 不加入无限自主循环、后台静默写入或无确认批量删除。
- 不重写现有聊天、Skill 选择和工具安装体系。
- 不删除内置工具或内置 Skill。
- 不承诺卸载工具依赖缓存；工具回滚只保证恢复原工具代码、元数据和可调用状态。

## 总体流程

```text
用户目标
  -> 意图与风险分类
  -> 结构化观察项目现场
  -> 生成有限步骤计划
  -> 预检全部动作
  -> 等待用户确认
  -> 创建事务快照
  -> 执行一个步骤
  -> 读取真实结果
  -> 验证步骤结果
  -> 继续 / 重新规划 / 回滚 / 停止
```

普通问答继续走现有聊天流程。只有需要产品内动作且能够映射到受控动作集的请求才进入有限自主流程。重新规划若产生新的写操作，必须再次等待确认。

## 组件边界

### 项目观察器

项目观察器生成 `JiziProjectObservation`，作为规划、预检和验证的统一事实来源。它包含：

- 当前画布、全部可编辑画布及只读运行副本的基本信息。
- 节点 ID、名称、类型、位置、Agent 引用、提示词摘要、模型引用、工具标签和输入输出格式。
- 画布连线的源节点与目标节点。
- Agent 库中的名称、描述、系统提示词、模型和工具配置。
- 自定义工具和内置工具的名称、来源及能力摘要。
- 最近一次运行中每个节点的状态、输出摘要、错误详情和工具结果摘要。
- Python 服务、模型、搜索源和 Skill 的可用状态。

模型只接收经过长度预算、脱敏和裁剪的观察文本。动作预检与结果验证使用完整结构对象，不能依赖裁剪文本。

### 计划解析与验证器

计划解析器将模型 JSON 转换为严格的判别联合类型。未知字段可以忽略，未知动作、空目标、非法模型引用、非法工具名和不完整参数必须拒绝。

计划验证器返回：

```ts
interface JiziPlanValidation {
  ok: boolean;
  normalizedSteps: MasterPlanStep[];
  issues: Array<{ stepIndex: number; code: string; message: string }>;
  risk: 'read' | 'write' | 'destructive';
  requiresConfirmation: boolean;
  requiresSecondConfirmation: boolean;
}
```

`destructive` 只用于删除画布和删除自定义工具。覆盖工具、批量修改和普通画布/Agent 修改属于 `write`。

### 动作事务器

事务器只执行通过预检并完成确认的动作。事务分为两类：

1. 本地状态事务：画布和 Agent 在执行前保存深拷贝快照；失败时一次性恢复。
2. 外部补偿事务：工具覆盖和删除前保存完整工具定义；失败时重新安装旧定义并刷新工具注册表。

事务结果必须区分：

```ts
interface JiziTransactionResult {
  ok: boolean;
  completedSteps: number;
  failedStep?: number;
  error?: string;
  rollback: 'not-needed' | 'succeeded' | 'partial' | 'failed';
  rollbackDetails: string[];
}
```

运行画布属于不可回滚的外部副作用，必须排在状态修改之后。若运行失败，状态修改是否回滚由计划声明的验收目标决定；默认回滚本轮配置修改，并保留运行错误记录作为诊断证据。

### 有限自主控制器

控制器使用显式状态机：

```ts
type JiziAutonomyState =
  | 'observing'
  | 'planning'
  | 'awaiting-confirmation'
  | 'awaiting-destructive-confirmation'
  | 'executing'
  | 'verifying'
  | 'replanning'
  | 'completed'
  | 'failed'
  | 'cancelled';
```

每次执行只运行一个已确认计划中的下一步。步骤完成后重新观察项目，并由确定性验证器优先判断结果；只有无法用本地规则验证的语义目标才允许模型辅助判断。模型判断必须返回证据引用，不能只返回成功布尔值。

控制器限制：

- 一个任务最多 8 个已执行步骤。
- 单个步骤最多修正 1 次。
- 整个任务最多重新规划 2 次。
- 连续两次没有项目状态变化时停止。
- 用户取消后不得继续排队动作。
- 任何新写操作或计划范围扩大都返回确认状态。

## 动作集扩展

在现有动作基础上新增：

- `update-agent`：修改 Agent 名称、描述、系统提示词、模型引用和工具标签。
- `update-node-agent-config`：修改当前画布节点的对应配置。
- `delete-canvas`：删除指定非运行副本画布，必须二次确认。
- `overwrite-tool`：覆盖自定义工具，禁止覆盖内置工具。
- `delete-tool`：删除自定义工具，必须二次确认，禁止删除内置工具。

所有动作使用稳定 ID 定位对象。用户语言中的名称只在规划阶段解析为 ID；执行阶段不得再次按模糊名称搜索。

## 诊断收紧

诊断类型改为：

```ts
type FailureCategory =
  | 'missing-tool'
  | 'tool-parameters'
  | 'node-configuration'
  | 'missing-input'
  | 'model-call'
  | 'network-or-service'
  | 'unknown';
```

只有 `missing-tool` 且具有明确报错证据时才能进入工具生成。JSON 解析失败、字段缺失、低置信度或证据为空时统一返回 `unknown`，展示原始错误并停止自动生成工具。

## 长期记忆

长期记忆升级为结构化记录：

```ts
interface JiziMemoryRecord {
  id: string;
  kind: 'profile' | 'preference' | 'resource';
  content: string;
  source: { sessionId?: string; messageId?: string; origin: 'conversation' | 'user-edit' | 'migration' };
  createdAt: number;
  updatedAt: number;
  confidence: number;
  scope: 'global' | 'project';
  expiresAt?: number;
  status: 'active' | 'superseded' | 'conflicted' | 'expired';
}
```

现有三个字符串数组迁移为 `origin: migration` 的活动记录，不丢失内容。新记忆写入前进行语义匹配；明确的新偏好覆盖旧偏好，疑似冲突进入 `conflicted`，不得同时作为有效背景注入。

## 深度搜索

搜索保持现有 Serper、Tavily 和 Brave 故障切换，在结果质检后增加正文读取层：

- 只允许 `http` 和 `https`。
- 阻止 localhost、私有地址、链路本地地址和非标准跳转，避免 SSRF。
- 限制响应大小、内容类型、重定向次数和请求时间。
- 提取正文并按用户问题选择相关片段。
- 来源分为官方、权威媒体、社区和未知。
- 关键事实优先要求两个独立来源；只有一个来源时明确标注。
- 正文读取失败时退回摘要，并在回答元数据中标记降级。

## 用户界面

沿用现有姬子确认卡片，不新建独立工作台。复杂任务卡片增加：

- 当前阶段和步骤进度。
- 计划步骤、风险级别和预检问题。
- 普通确认与删除二次确认。
- 每步真实执行结果和验证证据。
- 回滚状态及残留影响。
- 停止任务按钮。

界面不展示内部思维过程，只展示计划、使用的事实、动作结果和可解释的验证依据。

## 数据迁移与兼容

- 所有持久化 Store 增加版本号和显式迁移函数。
- 未识别的新字段由旧版本忽略；迁移失败时保留原始持久化值并回退到只读提示，不清空用户数据。
- 现有 `MasterAction` 继续可执行；旧的待确认动作不自动升级为自主任务。
- 内置 Skill 与用户 Skill 继续并存，升级不改变 Skill 存储目录。

## 测试策略

### 单元测试

- 观察器输出完整节点、连线、配置和运行证据，并正确脱敏裁剪。
- 计划解析拒绝未知动作、非法 ID、空参数和越权删除。
- 风险分类正确区分普通写入和删除。
- 诊断解析失败返回 `unknown`。
- 记忆迁移、覆盖、冲突和过期行为正确。
- 网页安全校验拒绝私有地址和危险重定向。

### 事务测试

- 多步计划中间失败后恢复画布和 Agent 快照。
- 工具覆盖失败后恢复原工具定义。
- 工具补偿失败时返回 `partial` 或 `failed`，不能谎报恢复成功。
- 删除动作未经过二次确认时无法执行。

### 状态机测试

- 成功任务完成观察、确认、执行和验证。
- 新写操作重新规划后再次等待确认。
- 超过步骤、修正或重新规划上限时停止。
- 用户取消后不再执行后续步骤。
- 无状态变化两次后停止并说明原因。

### 端到端评测

建立至少 60 个固定任务，覆盖画布编辑、Agent 配置、工具操作、失败恢复、搜索、记忆冲突、用户改口和取消回滚。发布门槛：

- 简单任务成功率不低于 95%。
- 多步任务成功率不低于 80%。
- 错误工具生成率低于 3%。
- 可回滚状态的回滚成功率为 100%。
- 未确认高风险写操作数量为 0。

## 实施顺序

1. 修复默认提示词冲突、诊断默认误判和结构化校验。
2. 建立结构化项目观察器并替换薄上下文。
3. 扩展动作、预检、风险分类、快照和补偿回滚。
4. 实现有限自主控制器并接入现有确认卡片。
5. 迁移长期记忆并实现冲突治理。
6. 增加安全正文读取和来源交叉验证。
7. 建立端到端智力评测并执行全量验证。

每个阶段必须保持应用可构建、现有测试通过，并提供该阶段新增行为的自动化测试。若某阶段暴露结构性阻塞，应停止继续叠加修改，将阻塞拆成明确清单，逐项解决后再进入下一阶段。
