## 六、关键架构决策（8 条）

1. **Tauri 替代 Electron** — 包体 5–15MB vs 100MB+，代价是 Rust 不会写就边查边写
2. **两层 Agent 模型** — 库定义 vs 画布实例快照，改定义不回写已落节点
3. **明文 JSON 存项目内** — 单用户便携优先，API Key 明文是权衡
4. **Python 服务单独跑** — WebView 调不了 openpyxl/PyMuPDF，FastAPI 跑 localhost:18081
5. **姬子能直接执行动作** — 8 类 MasterAction + 确认闭环，白名单 + 确认双险防控
6. **撤销历史双重裁剪** — 单画布 50 条 + 全局 200 条，按 seq 跨画布淘汰最旧
7. **两档自愈** — 补已注册标签零确认（无 RCE）；生成新工具人工审代码（RCE 硬底线）
8. **运行用只读快照** — 不直接改原画布，便于多次重跑对比

## 九、开发方法论（用 AI 的做法）

1. **清单做事实源** — `开发清单.md` + `施工清单.md` 先改再让 AI 写代码；代码比清单多且有用时反向改清单
2. **动作集 + 确认闭环** — 姬子能干 8 类活，每件都要"解析→列清单→用户确认→落盘"
3. **轻量校验优先** — 能复用既有工具就别造新基础设施（function-calling 那种大件等真需要再加）
4. **拆模块对抗文件膨胀** — `canvasStore.ts` 1026 → 645 行，`masterActions.ts` 573 → 9 行门面 + 子模块
5. **跨栈自检** — `npm lint/build` + `cargo check --offline` + `py_compile/unittest` 三套全绿才标完成
6. **保留失败的可能** — 不绕 `--no-verify`、不注释掉测试、不删报错代码
7. **留交接文档** — 交接注意事项、技术评审报告、我是怎么把这软件做出来的、模块化拆分规范
