## 一、项目是什么，我是谁

### 1.1 项目是什么

**多 Agent 协同工具**，给游戏测试人员用的桌面应用。核心功能：

- 在画布上摆多个 AI Agent 节点、连线、跑流水线
- 需求文档进来 → 测试用例出来 → Bug 报告出来
- 带一个叫"姬子"的总 Agent 助手，能用自然语言直接操作画布
- 节点失败时姬子能自动诊断、补工具、就地重跑

跑在 Windows 桌面上，分发成一个 `.exe` 安装包（NSIS/currentUser + python-dist 方案已建立，正式发版还缺 CI、签名和发版纪律）。

### 1.2 我是谁

- **职业**：4 年黑盒游戏测试。我懂的是测试流程、Bug 报告、用例设计，不是工程实现。
- **编程基础**：Python 懂类和对象，能读不能写大工程；C#/Unity 在自学，新手水平。
- **不会的**：TypeScript、React、Rust、Tauri、FastAPI、WebView——这软件用到的主栈，我开始时一样都不会。

### 1.3 为什么开始做

我自己测游戏的时候，每次写用例、整理 Bug 报告都是重复劳动，想用 AI 把这部分自动化。市面上的工具要么是纯网页跑不动本地文件，要么是单 Agent 串不出流水线。所以我决定自己做一个。

## 三、软件功能全清单

### 3.1 画布系统

- 多画布工作台：最多 20 个打开画布，页签创建/删除/重命名/双击重命名/保存/不保存关闭/只读运行快照
- 画布导航：缩放（65%–200%）+ 平移 + 适合屏幕 + 撤销栈（50 步，Ctrl+Z）
- 节点操作：拖拽入画布、删除、按 Ctrl 框选多选、Ctrl+C/V 复制粘贴（含连线）、递归折叠（含计数徽标）
- 连线系统：创建/删除、选中态颜色区分（默认蓝→选中橙加粗）、箭头同步变色
- 快捷操作：Ctrl+F 节点搜索黄框高亮 + 平移定位
- 状态栏：节点数/连线数/保存状态/运行进度条

### 3.2 Agent 系统（两层属性模型）

- **定义层**：库中编辑 Agent 模板（名称、描述、系统提示词、工具标签、LLM 选择）
- **实例层**：拖入画布即快照复制为独立节点，实例与定义完全解耦（改定义不回写已落节点）
- **操作**：创建（预置模板选择）、克隆（自动加「副本」）、删除（前置引用检查）、拖拽排序
- **导出/导入**：可移植 JSON 信封（剥离 id/时间戳、modelRef 置空留 modelHint）

### 3.3 工具系统（Python 服务 + 内置工具）

- **5 个用户工具**：`file`（读写删文件）、`docx`（Word 读写）、`excel`（表格操作）、`pdf-read`（PDF 解析）、`llm-calling`（LLM 代理）
- **工具管理**：动态注册、标签化、试运行验证、自定义工具生成与安装（需人工审阅完整代码确认）
- **路径沙箱**：`file` 工具允许根目录校验，用户选择外部输出目录时显式放行
- **工具迁移**：旧标签（file-read/file-write/file-delete）自动迁移到新名（file），存量数据无痛升级

### 3.4 Agent Runner（执行引擎）

- **执行模式**：顺序执行、依赖满足后并行执行（最多 4 个节点并发，启动错峰 80ms）、多画布批量并行（最多 6 个纳入，6 个并发运行）
- **失败隔离**：前序节点失败只影响其后续依赖，无依赖旁支继续；临时 LLM 失败节点级补跑一次
- **数据流转**：上游 JSON 优先流转（包含标题、摘要、完整数据），下游无上游时读手动文件/历史产物
- **schema 匹配**：上游输出格式不符下游输入时由 LLM 一次性转换，转换失败回落文本兜底
- **子图重跑**：失败节点 + 下游子图就地重跑，不新开画布，无法就地时回落整图重跑
- **限制**：单个输出文件 10MB、上传文件 2MB、历史数据选择（从 outputs/data.json 选先前结果）

### 3.5 总 Agent 姬子

- **对话闭环**：多会话聊天（100 条上限自动开延续会话继承摘要）、附件上传（图片/文档）、发送/取消/确认
- **联网搜索**：多厂商故障转移（Tavily→Brave→Serper）、前置上下文改写检索词、搜索质量判断、参考来源链接展示
- **记忆系统**：profile/preferences/resources 三类，支持管理与更稳的检索注入
- **自然语言操作**：LLM 结构化规划 + 多步确认执行，支持创建画布、加节点、连线、删节点、改输出格式、运行、重跑子图、重命名画布等动作
- **自愈闭环**：节点失败→编排层诊断→生成工具（人工审阅确认装依赖）→补已注册标签（零确认）→重跑
- **提示词优化**：文件导入式、prompt caching、人格瘦身 43%、当前时间注入、注入防御

### 3.6 模型配置中心

- 多厂商支持：预置 13 家（Anthropic、OpenAI、DeepSeek、Qwen、Kimi、GLM、MiniMax、MiMo、Gemini、Grok、SiliconFlow、OpenRouter、Together）+ 自定义
- 功能：通信测试（真实 chat）、获取模型列表、精选星标、能力标签（视觉/语音/长上下文）、密钥明文存储
- 协议适配：anthropic `/messages` + openai/gemini `/chat/completions` 三协议非流式

- **能力管理**：Skill 商店按内容自动分类，用户看到中文名和中文介绍，模型读取更完整的英文说明、能力清单和使用规则。
- **模型能力识别**：配置模型后自动推断长文、看图、音频等能力；旧配置会迁移重算，但仍保留人工修正空间。
- **任务与体检**：任务计划和一键体检移入姬子抽屉，外部只放简短状态和画布相关操作。

### 3.7 联网搜索

- 三家适配器归一化（Tavily、Brave Search、Serper）
- 标题栏搜索配置：多家 Key 管理、启停开关、优先级排序、即时验证按钮
- 发送前检索→AI 回复下方附参考来源
- 接入 `@tauri-apps/plugin-http` + capabilities 显式 allow 三家域名，桌面应用中三家均可用

### 3.8 报告中心

- 运行指标：总次数、成功率、产物数、平均耗时、失败/中止数、最近产出
- 画布/节点表现：过滤展示、历史产物筛选表格
- 产物管理：打开/定位/删除、产物摘要展示、已移除状态标注
- 自愈统计：补工具+安装工具计数入报告卡

### 3.9 数据持久化与便携

- 项目内明文 JSON：`data/multi-agent-*.json`（模型、搜索、Agent、画布、运行历史、会话、记忆）
- 按月份/日期/画布运行/节点产物分级输出：`outputs/YYYY-MM/YYYY-MM-DD/画布名/节点名/产物文件`
- 每个产物同时生成机器可读 `data.json` 信封（含节点元数据、摘要、结构化数据、原始回复）
- 6 个 Zustand store 持久化 + 语义化版本迁移脚本

### 3.10 应用生命周期

- 启动数据恢复：异常重启遗留运行状态自动标记为已中断
- 运行保护：运行中关闭窗口被拦截+提示、Tauri 主进程异常写 `crash.log`
- 环境配置器：自动检测 Python 3.10+、创建 venv、安装依赖
- Python 服务管理：启动/停止/自动重启（最多 3 次）、版本握手验证、后台端口复用

## 四、技术栈详解

### 4.1 前端（React + TypeScript）

**核心库**：
- React 19.2.7 + TypeScript 6.0.2 + Vite 8.1.1（构建工具）
- Zustand 5.0.14（状态管理，比 Redux 轻，API 简单）
- Ant Design 6.5.0（UI 组件库，组件多、文档全）
- @xyflow/react 12.11.1（React Flow，画节点连线图的库）
- Tauri API 2.11.1（前端调 Rust 命令的桥）
- OXLint 1.71.0（lint 工具，比 ESLint 快）

**为什么选这些**：
- Ant Design 是因为组件多、文档全，我能直接拼
- React Flow 是因为能画节点连线图，正好是画布需要的样子
- React 19 是当时最新版，我赌的——出问题就查
- Zustand 是因为比 Redux 简单，新手友好

### 4.2 桌面层（Rust + Tauri）

**核心库**：
- Tauri 2.11.3（桌面应用壳）
- Rust 1.77.2+（系统编程语言）
- serde 1.0（JSON 序列化/反序列化）
- tauri-plugin-http 2.5.9（HTTP 请求，绕过 CORS，rustls-tls+http2）

**Tauri 这一层做什么**：
- 启动/管理 Python 子进程
- 读写项目内 JSON 文件（持久化）
- Windows 资源管理器集成（打开文件、打开目录）
- HTTP 请求绕过 WebView CORS 限制（LLM API、搜索 API）
- 崩溃日志记录

**为什么不用 Electron**：
- 包体 5–15MB vs Electron 100MB+
- Rust 替代了 Chromium+Node
- **代价**：Rust 我一行都不会写，Tauri 那层所有命令都是边查边写的

### 4.3 Python 服务（FastAPI + 工具链）

**核心库**：
- FastAPI 0.115.12 + Uvicorn 0.34.3（Web 框架 + ASGI 服务器）
- openpyxl 3.1.5（Excel 读写）
- PyMuPDF 1.26.1 + pdfplumber 0.11.6（PDF 解析）
- python-docx（Word 文档读写）
- chardet 5.2.0（编码检测）
- requests 2.32.3（HTTP 代理 LLM API）
- Pydantic 2.11.4（数据验证）

**为什么有 Python 服务**：
这是被功能逼的。我要读 Excel、读 PDF、写 Word 文档——这些 TypeScript 在浏览器/WebView 里都做不了，得有原生代码。Tauri 的 Rust 层能做，但我不会 Rust 写这些库。Python 有 `openpyxl`、`PyMuPDF`、`python-docx`，现成的。

**架构**：FastAPI 跑在 localhost:18081，Tauri 拉起子进程，前端通过 `executeTool(toolName, params)` 调用 `POST /tools/{tool_name}/execute`。

### 4.4 三层架构

```
┌─────────────────────────────┐
│  前端（React + TypeScript） │  UI、状态、调度
└──────────────┬──────────────┘
               │ Tauri API（invoke 命令）
┌──────────────▼──────────────┐
│  桌面层（Rust + Tauri）      │  存储、系统调用、HTTP 代理
└──────────────┬──────────────┘
               │ HTTP（localhost:18081）
┌──────────────▼──────────────┐
│  Python 服务（FastAPI）      │  文件读写、Excel/PDF/Word 解析
└─────────────────────────────┘
```

三层架构不是我设计出来的，是我能力边界逼出来的——每层我都只能用最熟的工具。

## 五、不懂技术的我是怎么用 AI 做出来的

这部分是我最想留下来的——因为我用过 AI 写代码、也用过 AI 当顾问，这个项目让我摸清了什么时候该用哪种。

### 5.1 用 AI 的七条方法

#### 方法 1：清单做事实源，AI 做执行者

我维护两份清单：
- `开发清单.md`：92KB，功能规格 + 决策记录 + 变更记录表
- `施工清单.md`：51KB，按里程碑组织的施工勾选

**每轮迭代先改清单，再让 AI 按清单写代码**。这比"让 AI 自由发挥"稳得多。AI 不会跑偏，我也随时能复盘做到哪了。

**关键纪律**：如果 AI 发现代码实际比清单多、且是有用标准，**反向更新清单**——以技术为准。我之前曾经发现清单写的"4 个总 Agent 动作"，但代码里实际有 8 个——立刻改清单，不是坚持清单让 AI 改回去。

#### 方法 2：动作集 + 确认闭环控制 AI 的写权限

姬子能直接执行画布操作（创建画布、加节点、连线、运行），但被框死在 8 类动作里。动作集外的写操作（删画布、改配置、装工具）一律给手动步骤。

闭环是：**解析 → 列清单 → 用户文字确认 → 落盘**。让 AI 助手"能干活但不越权"。

自愈也分两档：
- **补已注册工具标签** → 零确认自动补 + 重跑。因为只改配置，无代码执行风险。
- **生成新工具** → 必须展示完整 Python 代码供人工审阅，确认后才装依赖。RCE 硬底线。

#### 方法 3：轻量校验优先，避开过度工程

M7 收尾的"按标签装配工具"这一项，我**没要** LLM function-calling 那套大件。只要运行前校验 toolTags 缺失就早失败。当天完成、当天验证。

同样：
- **历史数据选择**复用了既有的 `listOutputReports()` + `readFileInput()`，没造新基础设施。
- **进度条**直接用 `CanvasRunState` 已有的 total/completed/failed，只加一个 `skipped` 字段。

**"先复用、后造轮子"的克制**——是我在 4 天里把 M7→M11 全推完的关键。

#### 方法 4：拆模块对抗文件膨胀

我发现 AI 单次读不完整超过 1000 行的文件，会漏看尾部。所以我持续拆：
- `canvasStore.ts` 从 1026 行峰值拆到 645 行，逻辑分散到 `stores/canvas/*`（types/naming/derived/runRecords/runRecovery/selectors）
- `masterActions.ts` 从 573 行拆成 9 行门面 + `lib/masterActions/*` 细分模块
- `agentRunner.ts` 这次又拆出 `runGraph` 内核 + `ensureDataSources` / `ensureNodeCapabilities`

**拆模块不是为代码好看，是为了让 AI 下次能完整读懂。**

#### 方法 5：跨栈自检——三套都过才标记完成

每次改完代码我都会跑：
- 前端：`npm.cmd run lint` + `npm.cmd run build`
- Rust：`cargo check --offline`
- Python：`py_compile` + `unittest`

三套全绿才标记完成。**没为了"看起来全绿"绕过 `--no-verify` 或注释掉失败的测试**——这种克制比写新功能难，但我坚持下来了。

#### 方法 6：保留失败的可能

没为了"看起来全绿"绕过 `--no-verify`、没注释掉失败的测试、没删掉报错的代码。**这种克制比写新功能难。**

#### 方法 7：留交接文档

我给自己和未来的 AI 留了：
- `交接注意事项.md`：迁移清单、敏感信息、架构决策速记
- `技术评审报告.md`：评审记录
- `我是怎么把这软件做出来的.md`：个人视角的开发历程
- `全期开发进度统计.md`：客观统计视角的全期进度
- `docs/模块化拆分规范.md`：代码结构与拆分指南
- `MASTER_AGENT_PROMPT.md`：姬子人格文件
- `memory-bank/`（规划中）：RooFlow 风格的项目上下文

**这是给未来的自己留上下文**——避免每次都从零解释项目。

### 5.2 关键架构决策（每条附"为什么"）

#### 决策 1：两层 Agent 模型（定义 vs 实例）
- 库里编辑的是"定义"（模板），拖到画布上是"实例"（快照复制）。
- 改定义不回写已落节点，改节点不回写定义。
- **为什么**：避免"改了一个模板，所有正在用的实例都被污染"的协作歧义。我测游戏时遇到过类似问题——一个用例模板被改了，所有引用它的历史报告都对不上。

#### 决策 2：运行用只读快照，不直接改原画布
- 运行时创建画布快照 tab，原画布保持干净。
- **为什么**：便于多次重跑对比；快照隔离运行状态污染；快照 tab 关闭后自动清理，无垃圾副本堆积。

#### 决策 3：数据存项目内明文 JSON
- 不用数据库。所有数据在 `data/multi-agent-*.json`。
- **为什么**：MVP 单用户本地工具，数据便携性优先；JSON 易迭代、无需部署后端、离线可用。这个决定让我后来整个项目能拷到家里电脑继续做。API Key 明文是权衡——我知道不安全，但这是个人工具，便携优先。

#### 决策 4：Python 服务单独跑，不嵌进前端
- FastAPI 跑在 localhost:18081，Tauri 拉起子进程。
- **为什么**：Tauri WebView 调不了 openpyxl/PyMuPDF；HTTP 本地服务清晰分离关注点；Python 服务挂掉可独立重启，不影响主程序。

#### 决策 5：姬子能直接执行动作
- 8 类 MasterAction，LLM 输出受控 JSON 计划，用户确认后落盘。
- **为什么**：双险防控（白名单 + 确认），规划层与执行层解耦；操作失败不涉及代码生成，无安全隐患。

#### 决策 6：撤销历史双重裁剪
- 单画布 50 条软上限 + 全局 200 条硬上限，按 seq 跨画布淘汰最旧。
- **为什么**：长期使用画布越多历史越堆积，不裁剪会内存膨胀。跨画布淘汰是为了不让一个高频使用的画布把全局配额占满。

#### 决策 7：两档自愈
- 补已注册标签零确认 + 自动重跑；生成新工具必须人工审代码。
- **为什么**：补标签只改配置无 RCE 风险，零确认体验流畅；生成新工具涉及 Python 执行，RCE 硬底线不能松。

#### 决策 8：工具体系统一（v2.87）
- 5 条零散工具合并为 2 条主工具（`file` 带 action、`docx` 带 action），旧名为别名。
- **为什么**：之前 file-read/file-write/file-delete 是三个独立工具，用户配置时要在三个标签间切换；合并后一个标签覆盖多个操作，迁移函数 `toolTagMigration.ts` 让存量数据无痛升级。

### 5.3 我做的努力（容易被自己忽略的部分）

写这部分是因为我自己经常觉得"今天好像没干啥"，但回头看其实做了不少。给未来的自己留个证据：

#### 努力 1：维护两份清单的纪律
每次改完代码都要回填勾选和说明。这是体力活，让我随时能复盘"做到哪了"。**很多人做完功能不清不楚，我能随时说出来 M1–M11 每一项的状态，靠的就是这两份清单。**

#### 努力 2：留交接文档
`交接注意事项.md`、`技术评审报告.md`、`我是怎么把这软件做出来的.md`、`docs/模块化拆分规范.md`——**这是给未来的自己留上下文**，避免每次都从零解释项目。

#### 努力 3：提示词反复迭代
姬子的 `MASTER_AGENT_PROMPT.md` 从 v2.0 → v2.2 → v2.87 持续在改。每版都在补漏洞：
- v2.1 补"真实性约束""视野边界""注入防御"
- v2.2 加"执行动作集 + 确认闭环"
- v2.87 加"编排器 + 两档自愈"

**这不是一次性写完的，是看模型实际输出后反复收紧的**。比如模型老爱说"我记得你之前……"，但我没有跨会话记忆回放能力——发现后立刻在提示词里加"不得用'我记得你之前……'这种句式"。

#### 努力 4：三栈自检
Rust / Python / TypeScript 三套自检每次都跑全绿。**这不是天生的，是强迫自己养成习惯**——有过一次"前端 build 过了但 Python 编译挂了，我没跑就提交"的教训。

#### 努力 5：保留失败的可能
没为了"看起来全绿"绕过 `--no-verify`、没注释掉失败的测试、没删掉报错的代码。**这种克制比写新功能难。**

#### 努力 6：不渲染悲剧感
姬子这个角色来自崩坏3 无量塔姬子，原典里有"为后辈燃尽自己"的牺牲叙事。我在提示词里**明确禁止**这种渲染——"你不再为后辈燃尽自己，你是来干活的，目标是把活干完、把学生带顺"。这是我对这个工具的态度：它是来干活的，不是来演悲剧的。

### 5.4 失败和教训

#### 教训 1：跨栈提交前必须三套都跑
有过一次"前端 build 过了但 Python 编译挂了，我没跑就提交"的教训。从那以后强制自己三栈自检。

#### 教训 2：AI 单次读不完超长文件
`canvasStore.ts` 曾膨胀到 1026 行，AI 改它时经常漏看尾部，导致改了 A 处坏了 B 处。从那以后坚持拆模块——单文件不超过 600 行为佳。

#### 教训 3：模型会越权编造用户状态
模型老爱说"我记得你之前……""你当前有 3 个 Agent"——但姬子根本看不到 store 数据。必须在提示词里明确"视野边界"，列出信息来源清单。

#### 教训 4：网页 URL 抓取不能假设绕过 CORS
之前以为"接入 Tauri 后端即可绕过 CORS"是错误假设。浏览器纯预览阶段仍受限，必须 `plugin-http` + capabilities allow-list 三层齐全才能真联通。

#### 教训 5：列表返回必须稳定
Zustand selector 不能现算返回新数组/对象，否则白屏崩溃。派生值要用 `useMemo`。这是踩过坑后的硬规矩。

## 七、给未来的自己

如果你以后接手这个项目（不管是继续做还是推倒重来），记住三件事：

1. **先读清单，再读代码**。`开发清单.md` 是事实源，代码是清单的实现。如果代码和清单不一致，以代码为准——然后改清单。
2. **AI 能干活但不越权**。动作集 + 确认闭环是底线，别为了"方便"跳过确认。RCE 硬底线不能松。
3. **轻量校验优先**。能用既有工具就别造新基础设施。function-calling 那种大件，等真的需要了再加。

最后一个：**我什么都不会，但这不妨碍我把这软件做出来**。Rust 不会就边查边写，React 不会就读三遍文档，Tauri 那层所有命令都是第一次写。我不是天才，我只是坚持把每一步做扎实——清单维护扎实、提示词迭代扎实、跨栈自检扎实。

慢慢做，做扎实，做完。
