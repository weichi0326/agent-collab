## 2. 已确认问题（按优先级）

### P0-1：覆盖安装自定义工具失败时，会破坏旧版本文件

- **证据**：[`python/tools/installer.py:256`](python/tools/installer.py:256) 仅记录文件是否存在；[`python/tools/installer.py:258`](python/tools/installer.py:258) 在安装依赖、导入和注册前直接覆盖目标模块。异常分支 [`python/tools/installer.py:272`](python/tools/installer.py:272) 只删除“原本不存在”的文件，旧文件从不恢复。
- **复现路径**：已有 `foo.py` → 再安装同名 `foo` → 新依赖安装/模块导入失败 → 内存注册表仍可能指向旧函数，但磁盘 `foo.py` 已变成失败的新代码 → 重启服务后工具不可加载。
- **影响**：一次失败更新可让原本可用的自定义工具在下次重启后失效，且 UI 元数据与真实可执行代码可能不一致。
- **解决方案**：
  1. 先把新代码写到同目录唯一临时文件；在隔离模块名下执行编译/导入检查。
  2. 依赖安装成功、`execute` 校验成功后，再用原子替换提交目标文件。
  3. 若必须先替换才能验证，则在替换前读取旧字节，并在所有失败分支恢复旧文件和 `sys.modules`/注册表状态。
  4. 增加测试：覆盖同名工具时模拟 pip 失败、导入失败，断言旧文件内容、注册表和重启后的加载结果均不变。
- **验收标准**：任意安装失败后，旧工具仍能立即执行，服务重启后也仍可加载。

### P1-1：非法 `Content-Length` 会让请求体限制中间件返回 500，而不是正常拒绝

- **证据**：[`python/app.py:58`](python/app.py:58)-[`python/app.py:59`](python/app.py:59) 直接执行 `int(content_length)`，没有处理 `ValueError`。
- **影响**：客户端、代理或测试工具发送 `Content-Length: abc` 时会触发未处理异常。错误语义错误，也让服务健康性与日志出现噪声。
- **解决方案**：使用 `try/except (TypeError, ValueError)`；非法值返回 400。更稳妥的做法是用 Pydantic/Starlette 的流式大小限制，而不是只相信头部。
- **验收标准**：对缺失、合法、超限、非数字、负数 `Content-Length` 分别新增接口测试；服务均不返回 500。

### P1-2：请求大小上限可被无 `Content-Length` 的分块请求绕过

- **证据**：[`python/app.py:57`](python/app.py:57)-[`python/app.py:64`](python/app.py:64) 只在头部存在时检查大小，未在读取 body 时累计字节数。
- **影响**：60 MB 的声明上限不是实际约束；大 JSON 安装请求仍可能占用过多内存，拖慢同一个本地服务。
- **解决方案**：在 ASGI `receive` 包装器中累计 `http.request` body 长度，超过上限即终止；或为安装接口设置更小的模型字段上限（代码本身已有 20 KB 限制，应尽早拒绝大 body）。
- **验收标准**：使用 chunked 请求传入超过阈值的 body，稳定收到 413，进程内存不随请求体线性增长。

### P1-3：动态工具模块在一次加载流程中被执行两次

- **证据**：启动扫描在 [`python/tools/dynamic.py:93`](python/tools/dynamic.py:93)-[`python/tools/dynamic.py:95`](python/tools/dynamic.py:95) 依次 `import_module` 和 `reload`；安装流程在 [`python/tools/installer.py:264`](python/tools/installer.py:264)-[`python/tools/installer.py:265`](python/tools/installer.py:265) 也如此。
- **影响**：模块顶层初始化会执行两次。常规代码会造成重复日志/重复注册；一旦用户明确放行顶层副作用，可能重复写文件、发请求或创建资源。
- **解决方案**：新进程启动只 `import_module` 一次；更新已加载模块时才按需 `reload`。安装校验应使用临时模块名或显式 `importlib.util.spec_from_file_location`，避免对生产模块“先导入再重载”。
- **验收标准**：安装和重启各一次时，测试工具的导入计数均为 1；不再依赖顶层副作用放行也能正确更新。

### P1-4：运行时契约被复制在多处，容易出现前后端/打包配置漂移

- **证据**：
  - 服务版本在 [`python/tools/llm_calling.py:29`](python/tools/llm_calling.py:29) 与 [`app/src/lib/pythonClient.ts:50`](app/src/lib/pythonClient.ts:50) 各维护一份。
  - LLM 域名分别维护在 Python 白名单、[`app/src/lib/providers.ts`](app/src/lib/providers.ts)、[`app/src-tauri/tauri.conf.json`](app/src-tauri/tauri.conf.json) 与 [`app/src-tauri/capabilities/default.json`](app/src-tauri/capabilities/default.json)。
  - 工具元数据分别维护在 [`python/tools/dynamic.py:20`](python/tools/dynamic.py:20) 与 [`app/src/lib/toolRegistry.ts:25`](app/src/lib/toolRegistry.ts:25)。
- **影响**：新增模型厂商或工具时，可能出现“前端能选但后端拒绝”“开发环境可用但打包版本被 Tauri 拦截”“UI 文案与实际能力不符”。目前没有契约测试能阻止这类回归。
- **解决方案**：建立单一清单（建议 JSON/TS 源文件）并在构建时生成 Python、Tauri CSP/capability 片段；至少增加契约测试，比较服务版本、工具名、可访问域名和内部工具标记。
- **验收标准**：增加一个厂商/工具时只改一个来源；CI 能在任一派生配置漏更新时失败。

### P1-5：所谓“为打破循环而动态导入”并未实现分包，反而增加异步复杂度

- **证据**：生产构建明确警告 `canvasStore`、`modelStore`、`searchStore`、`toolRegistry`、`agentRunner/inputs`、`orchestratorStore` 的动态导入无效，因为它们同时被静态导入；相关说明见 [`app/src/stores/orchestratorStore.ts:143`](app/src/stores/orchestratorStore.ts:143)。构建结果主包为 1,566.55 kB（gzip 496.69 kB）。
- **影响**：首屏包偏大，动态导入没有产生预期的懒加载收益；异步加载、错误处理和循环依赖的排查成本却被保留。
- **解决方案**：
  1. 先用依赖图确认真实环路，抽出纯类型/纯接口层或事件接口层消除环路。
  2. 对必须同步使用的模块改回静态导入；只对页面级弹窗、报告中心等真正非首屏模块 `React.lazy`。
  3. 配置显式 chunk 策略并设置 bundle 预算（例如 gzip 主包 < 300 KB，按产品目标调整）。
- **验收标准**：构建不再出现 `INEFFECTIVE_DYNAMIC_IMPORT`；主入口包下降且功能测试覆盖诊断/重跑流程。

## 3. 清理与设计改进机会

### P2-1：`load_custom_tools` 的返回元数据当前没有消费者

- **证据**：[`python/tools/dynamic.py:79`](python/tools/dynamic.py:79) 声明返回 `list[dict]` 并在函数内累积 `metas`；唯一调用 [`python/tools/router.py:42`](python/tools/router.py:42) 丢弃返回值。工具元数据接口又自行调用 `read_registry()` 重建结果（[`python/tools/router.py:90`](python/tools/router.py:90) 起）。
- **判断**：这是已确认的冗余返回值/无效构造，不建议直接删除 `metas` 前先确认没有外部脚本将该函数作为公共 API 使用。
- **方案**：若启动阶段只需注册，改为返回 `None` 并移除 `metas`；若需要加载诊断，则把加载结果缓存为唯一元数据来源，避免每次 API 请求重读文件。

### P2-2：动态工具安装的“人工确认”与“允许顶层副作用”混在同一条风险链

- **证据**：路由公开 `allow_top_level_side_effects`（[`python/tools/router.py:70`](python/tools/router.py:70)），并在安装器中用它直接跳过顶层副作用检查（[`python/tools/installer.py:246`](python/tools/installer.py:246)）。
- **影响**：UI 上一次确认同时放开“安装依赖、写磁盘、导入执行代码”三类风险；后续难以审计用户到底批准了什么。
- **方案**：默认彻底移除该放行路径；如产品确有需求，改为显示风险摘要、一次性确认令牌、审计记录，并在受限子进程中做烟雾测试。此项需要产品/安全决策后再实施。

### P2-3：存储写入使用固定 `.tmp` 名称，存在同键并发写的竞态风险

- **证据**：[`app/src-tauri/src/storage.rs`](app/src-tauri/src/storage.rs) 的 `storage_set` 固定使用 `path.with_extension("tmp")` 后写入并 rename；前端 Zustand 持久化为异步写入（[`app/src/lib/tauriStorage.ts`](app/src/lib/tauriStorage.ts)），代码未见同键写队列。
- **判断**：这是**需压测确认**的竞态风险，不是本次已复现的故障。
- **方案**：每次写入用唯一临时文件（随机后缀），并按 storage key 串行化/“只保留最后一次写入”；增加并发更新同一 store 的压力测试。

### P2-4：仓库缺少可读的项目入口文档、CI 与已提交基线

- **证据**：根目录文件扫描未发现 README、AGENTS 或 `.github` CI；`git status --short` 显示全部项目文件未跟踪。
- **影响**：新 AI/新开发者无法可靠知道启动顺序、依赖版本、测试命令、发布流程和哪些文件不可提交；代码审查也没有可比较的基线。
- **方案**：在根目录补 `README.md`（架构图、启动、验证、数据目录、发布）、`AGENTS.md`（修改边界与命令）；建立最小 CI：前端 lint/test/build、Python compile/pytest、Rust cargo check；确认忽略规则后做首次提交。

## 4. 测试与配置缺口

1. **Python 测试无法执行**：`python/pytest.ini` 和 `python/tests/` 已存在，但 [`python/requirements.txt`](python/requirements.txt) 未声明 `pytest`，当前 venv 报 `No module named pytest`。新增 `requirements-dev.txt` 或在现有环境安装脚本中明确安装 pytest，并把 `python -m pytest -q` 放入 CI。
2. **跨端契约没有测试**：当前 42 个前端测试为纯 TS 逻辑；缺少“Rust 拉起 Python → token 鉴权 → `/tools/meta` → 工具执行”的最小集成测试。
3. **失败回滚没有测试**：至少覆盖 P0-1 的同名工具覆盖失败、动态模块导入一次、非法/分块请求体、版本不匹配自动重启。
4. **构建预算未门禁**：Vite 已警告入口包过大和无效动态导入，但构建仍成功。将警告转为 CI 可判定的预算检查，防止持续膨胀。

## 5. 推荐实施顺序（AI 可直接按此执行）

1. 建立 Git 基线、README/AGENTS 和 CI 骨架；不提交 `data/`、`outputs/`、`logs/`、venv、node_modules、python-dist。
2. 修复 P0-1，并先补失败回滚测试；这是防止用户工具丢失的最高优先级。
3. 修复 P1-1/P1-2，新增 FastAPI 接口测试，确保异常请求稳定失败而非 500 或占满内存。
4. 修复 P1-3，改造动态工具导入/更新流程并验证只执行一次。
5. 为 P1-4 引入清单或契约测试，先消除配置漂移风险再继续加新厂商/工具。
6. 处理 P1-5：先清除真实循环依赖，再做页面级拆包和 bundle 预算。
7. 最后处理 P2 项；其中 P2-2、P2-3 均需先确认产品与并发语义。

## 6. 大白话总结

| 问题 | 怎么修 | 不修会怎样 |
| --- | --- | --- |
| 更新一个自定义工具失败会把旧工具文件盖坏 | 新代码先在临时文件验证，全部成功再替换旧文件；失败就完整还原 | 用户今天还能用的工具，明天一重启就没了 |
| 请求头写错会把后端打成 500 | 把请求头当不可信输入，解析失败返回 400；按真实 body 限流 | 偶发请求就能把服务报错，排查时看起来像“系统自己坏了” |
| 自定义工具导入两次 | 新装/启动只导入一次，不要立刻 reload | 重复写文件、重复发请求、重复记账，问题还很难复现 |
| 前端、后端、打包配置各抄一份规则 | 改成一个清单生成，或至少自动比对 | 新增模型后可能出现“页面能点、运行报错”或“开发能用、安装包不能用” |
| 试图懒加载但实际上全塞进主包 | 先拆循环依赖，再真正按页面拆包 | 启动越来越慢，代码越来越绕，动态导入却没有收益 |
| Python 测试根本跑不起来 | 把 pytest 放进开发依赖和 CI | 后端改坏了也没人第一时间发现 |
