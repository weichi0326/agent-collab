## 一、结论摘要

项目当前可以通过前端、Python 和 Rust 全套自动化测试、lint、生产构建、Rust clippy 以及 Python 发布目录校验。复审没有发现仍处于开放状态的高严重度代码缺陷；此前确认的网络、文件路径、并发一致性、错误语义和数据清理问题均已按最小修改原则修复并补回归测试。

仍需后续产品设计的主要是两项“来源信息”问题，而不是当前功能已经失效：模型能力只保存最终布尔值，无法说明是自动推断还是用户设置；`api-tester` 的安全实现位于被 Git 忽略的本机自定义工具目录，不能随仓库可靠分发或升级其他机器上的旧副本。两项都在第四章单独说明。

原报告中的“严重 Token 时序攻击”“域名白名单未启用构成严重漏洞”“`file-write` 旧名已经失效”“删除工具后必然加载旧代码”“批处理文件应切换 UTF-8”等结论不成立或严重度明显失真，本版已移至“已排除或降级的旧结论”。

## 二、审计边界与判断前提

### 2.1 已核实范围

- React/TypeScript：画布运行、工具预检、模型能力、数据清理、持久化和报告相关路径。
- Rust/Tauri：Python 生命周期、路径命令、数据清理、退出保护和网络地址校验。
- Python/FastAPI：服务鉴权、动态工具安装/加载/删除、LLM 调用、网页读取和文件/Office 工具。
- 跨层一致性：工具名称、端口、路径范围、错误语义、模型能力和清理范围。
- 测试与构建：前端测试/构建/lint、Python pytest、Rust test/clippy。

本报告不再声称“逐文件深读全项目”或使用一个无法稳定复现的源文件总数。`python/tools/custom/`、运行数据、依赖目录和未提交文件会受忽略规则与本机状态影响，单一文件总数不足以证明审计覆盖率。

### 2.2 威胁模型

- 产品定位是 Windows、本机、单用户桌面应用，不按公网多租户服务评估风险。
- Python 服务默认监听 `127.0.0.1:18081`，由随机会话 Token 保护；手动开发启动时 Token 可以为空。
- 自定义工具代码通常需要用户确认后安装；当前工作副本还存在历史运行时工具，因此“需要确认”不能自动证明所有存量副本都经过当前版本的审查。
- 渲染进程通常被信任，但仍会处理模型输出、网页内容和用户导入内容；涉及任意本机文件能力时，应按渲染进程可能被攻陷的条件评估纵深防御。

### 2.3 严重度定义

| 级别 | 含义 |
|---|---|
| 高 | 可跨越明确安全边界，或在合理使用路径中造成敏感数据/系统级影响，应优先处理 |
| 中 | 已确认的错误行为、并发一致性问题，或可能明显误导用户和自动修复流程 |
| 低 | 有限场景下的错误、恢复性问题或维护成本，不阻塞当前主要功能 |
| 观察 | 风险取决于产品决策、部署方式或尚未复现的并发条件，需要记录但不能写成已确认漏洞 |

## 三、已确认并修复的发现（按原始严重度）

### HIGH-01 — 已加固：本机 `api-tester` 可请求任意地址

- **原问题**：本机登记的自定义工具直接请求调用方传入的 URL，可探测本机/内网；响应完整读入内存后才截断显示，无法限制下载量。
- **当前实现**：调用共享公网 URL 策略和固定 IP 网络层，拒绝私网、回环、链路本地、凭据 URL、代理和重定向；响应体流式读取上限为 2 MB，响应头总量上限为 64 KB。
- **验证**：工具自身测试覆盖私网拒绝、重定向拒绝、超大响应和正常 JSON；DNS 固定连接由 `safe_http` 测试覆盖。
- **剩余边界**：此工具属于被 `.gitignore` 排除的用户运行时副本，当前机器已安全，但仓库尚无可分发来源，见 OBS-06。

### HIGH-02 — 已修复：Tauri 路径命令缺少调用范围约束

- **原问题**：`open_path` 可打开任意存在路径，`read_text_file` 可读取渲染进程传入的任意小型文本文件；一旦渲染进程被控制，后端命令缺少第二道边界。
- **当前实现**：`open_path` 只允许规范化后的应用数据、输出和日志目录，并限制为报告/附件类型；脚本、快捷方式和可执行文件被拒绝。Skill 导入只读取原生文件选择器动态授权的普通 Markdown 文件，并按流最多读取 2 MB。
- **验证**：Rust 测试覆盖允许目录与相邻目录隔离、可执行扩展名拒绝及 Markdown 类型约束。
- **剩余边界**：仍缺 Windows 重解析点和真实原生对话框授权的端到端测试，属于测试深度，不是当前已知可利用路径。

### MEDIUM-01 — 已修复：工具安装与删除并发丢 registry 更新

- **原问题**：安装使用提交锁，删除却在锁外读取并重写 registry；两者并发时，后写入者可能覆盖先写入者的逻辑结果。
- **当前实现**：安装和删除共用 registry 提交锁；删除过程中若 registry 写入失败，会恢复模块文件，避免文件和登记状态分离。
- **验证**：安装、删除和失败回滚测试通过。

### MEDIUM-02 — 已修复：工具安装审计日志并发丢记录

- **原问题**：两个并发安装可能同时读取旧日志，再各自重写，导致其中一条审计记录消失。
- **当前实现**：读取、裁剪和原子重写由独立审计锁保护，不与 registry 锁混用。
- **验证**：并发追加测试确认两条记录均保留。

### MEDIUM-03 — 已修复：模型能力推断覆盖用户显式关闭

- **原问题**：旧合并逻辑使用逻辑或，模型名命中推断规则后，会把用户保存的 `false` 再次改成 `true`，影响附件发送和能力调度。
- **当前实现**：已有能力对象原样保留；只有新模型、旧数据确实缺少能力对象或用户明确点击“重新识别能力”时才推断。
- **验证**：测试覆盖已有 `false` 不被覆盖、新模型自动推断和手动重新识别。
- **剩余边界**：持久化数据仍不记录“这项值来自推断还是用户”，见 OBS-05。

### MEDIUM-04 — 已修复：`listTools` 把服务错误伪装成空列表

- **原问题**：后台未启动、鉴权错误和请求超时都返回 `[]`，上层因此误报“缺少工具”。
- **当前实现**：只有成功响应中的空数组表示没有工具；HTTP、网络和协议错误会明确抛出，上层区分“服务不可用”和“确实缺工具”。
- **验证**：测试覆盖 HTTP 错误、无效成功响应、正常空列表和正常工具列表。

### MEDIUM-05 — 已修复：SSRF 分类统一并绑定实际连接地址

- **已实施**：`security/ssrf-policy-cases.json` 定义两类明确策略并由 TypeScript、Python、Rust 测试共同消费。模型端点允许公网 HTTPS，以及显式 `localhost` / `127.0.0.1` 的本地 HTTP；网页读取允许公网 HTTP/HTTPS，但拒绝凭据、localhost、私网、链路本地和保留地址。
- **已修复绕过**：三端原先都会接受 `64:ff9b::a00:1`，它是 NAT64 映射的私网 `10.0.0.1`。当前实现会解码 NAT64 内嵌 IPv4 并按 IPv4 规则拒绝，同时允许映射公网地址的 NAT64 用例。
- **一致性边界**：`python/network_policy.py` 成为 Python 共享实现，`web_reader_policy.py` 只重导出内容读取策略。`python/safe_http.py` 在每次实际请求中只解析一次，拒绝答案中任何非公网地址，并让连接池直接连接已验证 IP；多地址回退也只使用同一批已验证结果，不再次解析域名。
- **TLS 与路由身份**：固定 IP 不改变 HTTP `Host`；HTTPS 同时把 TLS SNI 和证书主机名校验保持为原域名。环境代理被禁用，显式代理和自动重定向被拒绝；网页读取仍按原产品行为手动处理重定向，但每一跳都会重新解析、校验并绑定目标。
- **统一出口**：网页读取、`llm-calling` 的对话和模型列表、以及本机 `api-tester` 都使用固定 IP 网络层。`app/src/lib/llmClient.ts` 不再通过 Tauri HTTP 插件直连外部模型接口，避免前端校验后由插件再次解析。
- **验证证据**：单元测试模拟“第一次解析为公网、第二次本可变为回环”的序列，断言实际只解析一次并连接公网 IP；另覆盖混合公网/私网答案拒绝、已验证多地址回退、代理/重定向拒绝、localhost 只允许回环，以及真实本地 HTTP 连接的 Host 保留。
- **测试边界**：尚未在可控 DNS 服务器和真实公网 TLS 证书环境中做端到端对抗测试；这属于验证深度缺口，不再是当前实现仍会二次解析的已知设计缺陷。三个搜索 API 使用代码内硬编码厂商域名，不接收用户 URL，本轮未迁移其直连实现。

### LOW-01 — 已修复：画布建议把已有 URL 输入误判为无输入

- **原问题**：代码检查不存在的复数字段 `dataSourceUrls`，导致已经填写 URL 的入口节点仍被列为“可能缺输入”。
- **当前实现**：直接使用 `AgentNodeData`，按文件、URL、历史三种数据源模式检查各自真实字段，并复用公共节点名称函数。
- **验证**：测试覆盖每种模式的已配置和未配置状态。

### LOW-02 — 已修复：文件写入统计不是实际字节数

- **原问题**：二进制模式返回 base64 字符数，非 ASCII 文本返回字符数，与实际落盘字节数不一致。
- **当前实现**：为兼容现有调用方保留 `written` 字段，但其值统一为实际编码后或解码后的字节数。
- **验证**：测试覆盖中文文本和 base64 二进制写入。

### LOW-03 — 已修复：`python_status` 查询会自动重启服务

- **原问题**：读取状态具有启动副作用，服务重启时机受 UI 轮询影响，难以解释和测试。
- **当前实现**：`python_status` 只查询；应用启动和任务执行前仍分别通过 setup 与 `ensureCompatiblePythonService` 显式启动或恢复服务。
- **验证**：Rust 测试确认无子进程时只返回 `stopped`，不会创建进程。

### LOW-04 — 已修复：关闭保护在前端失联后无法恢复

- **原问题**：单个布尔值一旦未被清理，窗口会持续阻止正常关闭。
- **当前实现**：Rust 使用两分钟租约，前端在任务活动期间每 30 秒续约，正常结束立即释放；前端失联后租约自动过期。
- **验证**：Rust 测试覆盖续约、释放和超时。

### LOW-05 — 已修复：Zustand 水合回调直接修改状态对象

- **原问题**：水合完成回调绕过 store 的更新路径直接赋值，订阅者能否及时看到诊断会话依赖时序。
- **当前实现**：诊断固定会话在持久化迁移返回值中补齐，不再直接修改已创建状态对象。
- **验证**：测试覆盖旧持久化数据补齐且保留既有会话。

### MEDIUM-06 — 已修复：模型 Token 上限前后端不一致

- **原问题**：前端配置允许最大 30000，但 Python 后台静默截断到 16384。用户看到的是 30000，真正发给模型的却更少，长回答可能无故变短，且界面没有错误提示。
- **当前实现**：Python `MAX_TOKENS_LIMIT` 与前端统一为 30000；`LLM_CALLING_VERSION` 和 `EXPECTED_PYTHON_SERVICE_VERSION` 同步升级为 `2026-07-22.code-audit`，确保运行中的旧后台被识别并重启。
- **验证**：Python 边界测试和前后端契约测试通过。

### LOW-06 — 已修复：Unicode 与 Windows 文件名边界

- **原问题**：旧截断按 UTF-16 单元处理，在长度边界可能切断 emoji 等代理对；Agent 文件名还没有删除 Windows 不接受的末尾点。
- **当前实现**：新增共享 `sanitizePathSegment`，画布输出目录与 Agent 产物文件名都按 Unicode 字符截断，并统一清理控制字符、非法字符、末尾点和空格。
- **验证**：测试覆盖第 48 个字符为 emoji 和末尾点两种边界。

## 四、需要产品或架构确认的风险

### OBS-01 — 已修复：工具与 Skill 清理范围已拆分

- **原问题**：`tools_app_data` 同时包含工具配置和整个 `app_data_dir`，会连带删除 `skills/` 及未来增加的未知应用数据。
- **当前实现**：为兼容现有命令协议保留 `tools_app_data` ID，但范围收窄为工具配置文件和 `app_data_dir/python-tools`；新增独立的 `user_skills` 分类，只清理 `app_data_dir/skills`。两项均为可选、重要、默认不选中。
- **用户提示**：清理卡片和重要数据二次确认均显示具体影响。自定义工具清理会使已安装或生成的工具及依赖不可用，但不影响内置工具；用户 Skill 清理会删除创建、导入和覆盖内容，内置 Skill 恢复默认。
- **回归边界**：Rust 测试明确验证清理工具会保留 Skill 和未知应用数据，清理 Skill 会保留工具及其依赖。

### OBS-02 — 用户工具依赖目录位于 `sys.path` 首位

- **证据**：`python/tools/dynamic.py:127-140` 把用户 `site-packages` 插入 `sys.path[0]`。
- **判断**：依赖包理论上可以影响后续模块导入，但自定义工具本身已经是经用户批准、可在 Python 进程内执行的代码；这不是额外获得代码执行权限的独立漏洞。
- **建议**：把它记录为自定义工具信任模型的一部分。若未来允许无人工确认安装第三方依赖，再考虑隔离进程、固定哈希和依赖来源策略。

### OBS-03 — Excel 写操作没有文件级锁

- **证据**：`python/tools/excel_tool.py` 多处执行 `load_workbook` 后修改并 `save`，没有文件锁。
- **判断**：在当前单 worker、阻塞式 async 执行路径中，请求通常被事件循环串行化，本轮没有复现两个请求同时覆盖同一工作簿。因此不能列为已确认并发 Bug。
- **建议**：如果未来把 Office 操作放入线程池、启用多 worker 或允许多个进程写同一文件，应先加入同路径互斥锁和原子替换策略。

### OBS-04 — 模型/网页网络策略是否允许任意公网域名

- **证据**：README 明确支持“自定义公网 HTTPS 端点”，共享模型网络策略也按此实现。
- **判断**：允许任意公网 HTTPS 是当前产品能力，不是白名单绕过。旧的未使用域名白名单和误导注释已删除；除非产品明确取消自定义端点，否则不应恢复强制厂商白名单。

### OBS-05 — 模型能力缺少来源信息

- **大白话解释**：现在只保存“支持/不支持视觉、音频、长上下文”三个结果，不保存这个结果是软件根据模型名猜的，还是用户亲手改的。
- **当前影响**：本轮已保证保存过的值不会再被自动推断覆盖，所以眼前不会把用户关闭的能力擅自打开；但将来推断规则升级、模型改名或做设置同步时，系统无法判断哪些值可以自动更新、哪些必须尊重用户。
- **可能后果**：要么新规则不敢更新旧数据，能力识别长期过时；要么贸然更新，覆盖用户选择。它是数据模型的可维护性问题，不是当前会导致崩溃或泄密的 Bug。
- **后续建议**：需要迁移持久化结构时，再保存 `source: inferred | user` 或拆成 `inferredCaps + overrides`；不为这一项单独做高风险数据迁移。

### OBS-06 — `api-tester` 缺少可分发安全来源

- **大白话解释**：当前机器里的 `api-tester` 已经修安全了，但它放在 Git 明确忽略的“用户自定义工具”目录。换一台机器安装项目时，这份修好的代码不会自动带过去；旧机器上的历史副本也不会随仓库升级自动替换。
- **当前影响**：本机副本已有防护，仓库主程序也没有因此失效。风险在于多机器发布和后续升级：不能保证每台机器拿到同一份安全实现。
- **后续建议**：如果它要成为正式功能，应放入受版本控制的内置工具或安装模板，并给旧用户副本设计版本检测、迁移或重新确认流程；如果只作为个人工具，则应继续明确标记为本机自定义内容。

## 五、清理与解耦机会

### 5.1 可安全安排的小型清理

1. 统一 Rust 项目根目录定位。当前 `lib.rs`、`storage.rs` 和 `python_manager.rs` 仍有三套近似逻辑，但返回类型和调用语境不同，适合抽取小型公共模块，不需要大规模重构。
2. 继续拆分主入口中真正低频且边界清晰的功能；当前主包虽已缩小约 14%，仍超过 Vite 的 500 KB 警告线。

### 5.2 需要先确认迁移状态的重构

1. `memory` 与 `memoryRecords` 仍并行存在。旧 `memory` 仍被 `jiziMemory.ts` 的提示构建和相关逻辑消费，不能直接删除；应先迁移所有读取方，再设置持久化版本迁移和回滚策略。
2. 模型能力来源需要持久化迁移，不能只给现有布尔值加一个默认来源，否则会错误标记历史用户选择。
3. `api-tester` 若转为内置工具，需要决定如何处理同名用户副本、旧版本迁移和用户自定义代码信任边界。
4. 端口 `18081` 在 Rust 与 TypeScript 中重复。提取共享配置前需处理开发预览、打包资源和服务启动顺序，不建议为了一个常量引入运行时配置复杂度。

## 六、已排除或降级的旧结论

| 旧结论 | 核实结果 |
|---|---|
| Service Token 使用 `!=` 是严重时序攻击 | 可用 `hmac.compare_digest` 做低成本加固，但 localhost、随机会话 Token 和网络抖动使其不应定为严重漏洞 |
| `_ALLOWED_DOMAINS` 未使用导致白名单绕过 | 不成立；产品明确允许自定义公网 HTTPS。真实问题是死常量和误导性注释 |
| `load_custom_tools` 可通过模块名导入目录外任意模块 | 未找到成立的导入路径。增加标识符校验可防损坏 registry，但不能写成已确认任意模块导入漏洞 |
| `canvasAdvisor` 的缺输入建议永远不触发 | 不成立；真实影响是 URL 已存在时仍触发缺输入误报 |
| `remove_tool` 未清 `sys.modules` 导致重装使用旧代码 | 未成立。安装会注册隔离加载得到的新函数，进程重启时 `reload` 也会读取新文件；清理缓存可做卫生改进 |
| `canvasTransfer` 使用失效的 `file-write` | 不成立；`python/tools/router.py:24-37` 明确保留并注册兼容别名 |
| Excel 已确认存在并发覆盖 | 未复现；当前执行路径通常串行。仅在执行模型改变后成为明确风险 |
| `tools_app_data` 无提示误删 Skill | 不准确；UI 已标为重要的“工具与用户扩展数据”并二次确认。剩余问题是范围说明与分类粒度 |
| `环境配置器.bat` 应加入 `chcp 65001` | 不成立；README 明确规定 `.bat` 使用 GBK + CRLF，改成 UTF-8 code page 可能造成反向乱码 |
| Unix `.spawn()` 不 `wait()` 是当前产品缺陷 | 项目明确仅支持 Windows；可作为未来跨平台移植注意项，不进入当前修复计划 |
| `safeFileName` 与 `safeOutputPathSegment` 完全重复 | 不准确；核心字符替换相似，但长度、fallback 和尾部规则不同 |
| 所有存储都不存在数据转换损耗、所有启动脚本都无竞态 | 证据不足的宽泛正面结论，删除，不作为审计保证 |

## 八、建议执行顺序

### 第一批：补齐分发和数据来源设计

1. 决定 `api-tester` 是正式内置工具还是个人自定义工具；若正式分发，补受版本控制的安全来源和旧副本迁移策略。
2. 在下一次模型配置持久化版本迁移时，增加能力来源或用户覆盖信息；当前无需为此单独迁移。

### 第二批：补端到端边界测试

3. 在真实 Windows 环境覆盖路径重解析点、原生文件选择授权和报告打开流程。
4. 在可控 DNS 与真实 TLS 证书环境验证固定 IP 连接、SNI、证书主机名和 DNS 变化。

6. 如未来要求 Python 崩溃后无需用户触发任务即可恢复，再单独引入 Rust watchdog；不要重新给状态查询增加副作用。
7. 若未来取消自定义工具的人工确认，再引入进程级隔离、依赖固定哈希和来源限制。

### 第四批：维护性改进

8. 后续新增网络工具必须复用共享策略和固定 IP 出口，或提供同等级连接绑定测试。
9. 完成记忆格式迁移后再移除旧结构；在此之前不把兼容读取当作死代码。

11. 补 CI，使测试、lint、build、格式检查和 clippy 成为可重复的合并门禁。

## 九、最终判断

当前复审没有发现仍开放的高严重度代码缺陷。已确认问题集中在可控的网络/文件边界、并发一致性、错误语义和少量数据边界，本轮均已修复并有自动化测试。不要按旧报告机械启用厂商域名白名单、移除兼容工具名或修改批处理编码，这些做法会破坏现有产品能力。

现在剩下的是两项需要产品决策的设计债、若干端到端测试缺口、主包继续拆分和 CI 建设。它们值得继续做，但不应再描述成软件当前存在大量严重漏洞。
